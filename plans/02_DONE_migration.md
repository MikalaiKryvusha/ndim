# План 02 — Уважительная миграция данных 1.x → 2.0

> Перенос данных **299 реальных людей** из модели NDim Space 1.x в модель 2.0.
> Проект модели — `researches/04`, соответствие коллекций — там же §5, слепок 1.x — `researches/02`.
>
> ⚠️ Это работа с **боевыми данными живых людей**. Документ — строгий контракт, а не рекомендации.
> Заведён 2026-07-10.

---

## Что значит «уважительная» (требования владельца)

Из `researches/02` и интервью №002:

1. **Без потерь.** Ни один человек не должен ничего заполнять заново.
2. **Обратимо.** Старые коллекции не трогаем, пока миграция не подтверждена. Всегда можно откатиться.
3. **Сначала на копии.** Скрипт отлаживается на копии боевой базы, не на боевой.
4. **Статус-кво видимости.** Всё остаётся видимым всем, как было в 1.x (Т7). Ничего не раскрывается
   сверх прежнего и ничего не прячется молча.
5. **Паритет чисел.** Пересчёт связей после миграции обязан дать те же значения. Ядро это гарантирует
   (тесты `src/lib/similarity/`), но сверка всё равно нужна.

---

## Ключевой технический риск: связь по полю, а не по ключу

В 1.x документы связаны **не** по ключу:

| Коллекция | Ключ документа | Связь с пользователем |
|-----------|----------------|------------------------|
| `users/{userId}` | `userId` = `auth.uid` | ключ **и есть** uid |
| `ndimids/{ndimId}` | `ndimId` — отдельный id | по **полю** `owner_uid` |
| `relations/{relation}` | `relation` — отдельный id | по **полю** `owner_uid` |

**Поэтому группировать надо по `owner_uid`, а не по ключу документа.** И прежде чем что-либо
переносить, обязательно проверить инвариант:

> **У каждого `owner_uid` — ровно один `ndimids` и не больше одного `relations`.**

Если у кого-то их несколько (дубликаты, брошенные черновики) — миграция **останавливается** и зовёт
человека. Молча выбрать «какой-нибудь» документ с оценками человека недопустимо: это его данные.

## Соответствие коллекций (из `researches/04` §5)

| 1.x | 2.0 | Как |
|-----|-----|-----|
| `users/{uid}` (плоский профиль) | `users/{uid}` (корень) + `users/{uid}/profile/everyone` | профиль → бакет `everyone`; `visibility` = всё `everyone` (`migratedUserVisibility()`) |
| `users/{uid}.email` | — | не переносим: email живёт в Firebase Auth |
| `users/{uid}/privat/` | — | удаляем (пустая заглушка) |
| `ndimids/{*}` where `owner_uid==uid` | `points/{uid}` + `points/{uid}/dims/{dimId}` | каждая ось `user_dims[dimId]` → документ `{ value }` |
| `ndimids/{*}.sync_status` | `points/{uid}.dirty` | `'synchronized'` → `false`, иначе → `true` |
| `relations/{*}` where `owner_uid==uid`, поля `a…t` | `relations/{uid}.top[]` с именами | через `decodeLegacyRelation` (`src/lib/similarity/legacy.ts`) |
| `dims/{id}` | `dims/{id}` | без изменений |
| `suggestions/{id}` | `suggestions/{id}` | `author` (email) → `authorUid` — **нужна карта email→uid** (см. риск ниже) |
| `admins/{id}` | custom claims | не документы, а `admin: true` в токене |
| — | `friendships/`, `groups/`, `audience/` | при миграции **пусты**: люди заведут сами |

## Порядок работ

### Шаг 0 — Доступ и копия (только человек) 🔲

Агент не имеет права трогать боевой проект без явного разрешения. Оформляется **домашкой**:
1. Владелец делает экспорт боевого Firestore (`gcloud firestore export`) в bucket.
2. Импортирует его в **отдельный** проект-песочницу (или отдельную базу).
3. Даёт агенту `projectId` песочницы.

Вся отладка — на песочнице. Боевой проект до финального go/no-go остаётся нетронутым.

### Шаг 1 — Инвентаризация 🔲

Скрипт `tools/migrate/01-inventory.mjs` (Admin SDK, **только чтение**):
- сколько документов в каждой коллекции;
- **проверка инварианта**: у каждого `owner_uid` один `ndimids` и ≤1 `relations`; список нарушителей;
- сколько уникальных `dimId` встречается в `user_dims` всех людей, и все ли они есть в `dims`;
- карта `email → uid` из `users` — для миграции `suggestions.author`;
- сироты: `ndimids`/`relations` без соответствующего `users`; `suggestions` с неизвестным email.

Отчёт — в `researches/` (не в код). Пока отчёт не чист или нарушители не разобраны с человеком —
дальше не идём.

### Шаг 2 — Трансформация как чистая функция 🔲

`src/lib/migration/transform.ts` — **чистые** функции 1.x-документ → 2.0-документ(ы), без Firestore:
- `migrateProfile(user)` → `{ root, everyoneBucket }`;
- `migratePoint(ndimid)` → `{ point, dims: Map<dimId, {value}> }`;
- `migrateRelations(relationsDoc)` → `RelationsDoc` (через `decodeLegacyRelation`);
- `migrateSuggestion(sugg, emailToUid)` → `SuggestionDoc | needsHuman`.

Покрыть тестами на **синтетических** документах 1.x (не на боевых ПДн). Проверять:
- границы значений проходят (те же, что в `schema.ts`);
- оценки переносятся все до одной; ни одна ось не теряется и не дублируется;
- `visibility` = `migratedUserVisibility()` (всё `everyone`) — не `newUserVisibility()`;
- идемпотентность на уровне функции: повторный прогон даёт тот же результат.

### Шаг 3 — Прогон на песочнице 🔲

`tools/migrate/02-run.mjs` (Admin SDK, идемпотентный):
- пишет 2.0-документы **рядом**, не удаляя 1.x;
- батчами; при сбое можно перезапустить — идемпотентность по ключу документа;
- ведёт журнал: что создано, что пропущено, что ушло человеку.

### Шаг 4 — Сверка 🔲

`tools/migrate/03-verify.mjs`:
- число людей 1.x == число `users` 2.0 с непустым `profile/everyone`;
- сумма оценок каждого человека в 1.x == сумма в `points/{uid}/dims/*`;
- **пересчёт связей** новым ядром на мигрированных `points` → сравнить с тем, что расшифровано из
  старых `relations`. Должны совпасть (паритет проверен, но данные реальные — доверяй, но проверяй);
- ни одного потерянного `dimId`.

Расхождение — стоп, разбор с человеком.

### Шаг 5 — Боевой прогон (только человек, go/no-go) 🔲

После зелёной сверки на песочнице — владелец подтверждает, и те же скрипты идут по боевой базе.
Старые коллекции **не удаляются**. Наблюдаем несколько дней (люди заходят, видят свои профили и связи).

### Шаг 6 — Уборка 🔲 (недели спустя, отдельным решением)

Только после того как 2.0 доказал, что работает: `users/{id}/privat/`, старые `ndimids`, старые
`relations` архивируются экспортом и удаляются. Отдельная домашка, отдельное go/no-go.

## Риски

| Риск | Уровень | Что делаем |
|------|---------|-----------|
| Несколько `ndimids` на один `uid` | 🔴 высокий | шаг 1 останавливает миграцию, зовёт человека — не выбираем молча |
| `suggestions.author` — email, а не uid; email мог смениться | 🟠 | карта email→uid; неразрешённые заявки → человеку (их мало, авторы известны) |
| Сирота: `ndimids`/`relations` без `users` | 🟠 | шаг 1 составляет список; решает человек (удалить или восстановить профиль) |
| Оценки — самое ценное, что заполнили люди | 🔴 высокий | сверка суммой по каждому человеку (шаг 4); старое не удаляется до §6 |
| Пересчёт даст не те числа | 🟡 | паритет ядра в тестах + сверка на реальных данных (шаг 4) |
| Скрипт запустили на боевой вместо песочницы | 🔴 высокий | `projectId` песочницы — обязательный аргумент; боевой требует явного флага `--production` |
| 31 февраля и подобные даты 1.x | 🟢 низкий | границы 1.x приняты как есть (`schema.ts`); `isRealDate` только предупреждает |

## Когда это делать

Миграция нужна **перед публичным запуском 2.0**, не раньше. Пока фронтенда нет, торопиться некуда:
план зафиксирован, чистые функции трансформации можно написать и покрыть тестами в любой момент
(они не требуют боевого доступа), а прогон — когда 2.0 будет готов принять людей.

Приоритет сейчас (см. `MASTER_PLAN.md`): фронтенд важнее, потому что без него мигрировать некуда.
Поэтому следующий шаг после этого плана — **инициализация SvelteKit**, а не скрипт миграции.
