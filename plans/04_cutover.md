### 🔴 Два дефекта, которые вскрыл именно боевой выкат

Оба починены по ходу; оба — уроки в `EXPERIENCE.md`.

1. **Шаг 5 зависел от данных, которые затирает шаг 2** (EXP-0039). `04-admin-claim.mjs` строил
   карту `email → uid` по коллекции `users`, но миграция **перезаписывает** `users/{uid}` схемой
   2.0, где поля `email` нет (почта — ПДн, в корень не выносится). Итог: «2 почты → **0 uid**».
   **Починено:** почта разрешается в uid через **Firebase Auth** (`getUserByEmail`) — Auth
   переживает миграцию и там же живёт сам claim. Порядок шагов 2↔5 больше не имеет значения.
2. **Dockerfile отстал от кода** (EXP-0040). Копировал только `similarity.ts`, а `index.mjs`
   импортирует ещё и `model/stats.ts` → боевой контейнер **упал в цикле перезапуска**
   (`ERR_MODULE_NOT_FOUND`). **Починено:** копируем директориями (`src/lib/similarity/` +
   `src/lib/model/`). **Стенд поймать это не мог** — локально весь репозиторий лежит на диске.
   Отсюда правило: перед постановкой службой прогоняй **один цикл** (`--once`) и читай логи.
   Отсюда же родилась `ideas/13` (инсталлятор сервера с самопроверкой).

## Чего ждать от cutover

**1.x и 2.0 делят одну базу Firestore.** Правила 2.0 запрещают старые коллекции
(`ndimids`, `admins`), поэтому в момент выката правил приложение 1.x перестаёт работать.
Значит, порядок шагов — не вкусовщина, а способ не оставить людей без продукта.

## Сценарий (выполнять подряд, ~20 минут)

### Шаг 1. Свежая копия (страховка «на сейчас»)

```powershell
gcloud firestore export gs://ndim-space-backup/<сегодня> --project=ndim-space
```
Дождаться `SUCCESSFUL`: `gcloud firestore operations list --project=ndim-space --limit=1`.

### Шаг 2. Миграция боевых данных

```bash
node tools/migrate/02-run.mjs --production
node tools/migrate/03-verify.mjs --production
```
- Скрипт **только создаёт** документы 2.0 (`users/*/profile/everyone`, `points/*`, `suggestions/*`).
  Коллекции 1.x он не трогает — 1.x продолжает работать.
- **Связи не мигрируются**: они производные, их пересчитает сервер синхронизации.
- Сверка обязана быть чистой. Не чистая — **СТОП**, дальше не идём.

### Шаг 3. Сборка и выкат приложения на оба сайта

```bash
npm run build
npx firebase deploy --only hosting --project ndim-space
```
Выкат идёт и на `ndimspace.app`, и на `ndim-space.web.app` — люди 1.x попадают в 2.0.

### Шаг 4. Правила и индексы (после этого 1.x мёртв, 2.0 жив)

```bash
npx firebase deploy --only firestore:rules,firestore:indexes --project ndim-space
```
Между шагом 3 и шагом 4 — окно в пару минут, когда 2.0 уже отдаётся, а правила ещё старые:
человек увидит «Не удалось загрузить». Поэтому шаги 3 и 4 идут подряд, без перерыва.

### Шаг 5. Админ — это claim, а не список email (в 1.x был `admins/`)

```bash
node tools/migrate/04-admin-claim.mjs   # ставит admin: true владельцу
```

### Шаг 6. Сервер синхронизации в бой

```bash
docker build -f calculator/Dockerfile -t ndim-calculator .
docker run -d --restart unless-stopped \
  -v "$(pwd)/calculator/secrets:/secrets:ro" \
  -e GOOGLE_APPLICATION_CREDENTIALS=/secrets/sa.json \
  -e FIREBASE_PROJECT_ID=ndim-space \
  -e CALC_INTERVAL_SECONDS=3600 \
  --name ndim-calculator ndim-calculator
```
Первый цикл пересчитает связи всех 93 людей с непустым NDim ID и запишет статистику
Пространства. Проверить: `/space` → «Текущее состояние: Работает».

### Шаг 7. Проверка руками (владелец) — **обязательно**

1. **Вход через Google** — автотестом не закрыть (эмулятор Auth открывает интерактивное окно).
2. Вход по ссылке из письма **своей старой почтой 1.x** → в профиле его измерения и связи.
3. Гостевой режим с лендинга → оценка → «Сохранить результаты».
