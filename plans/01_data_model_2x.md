## Порядок работ

Снизу вверх: сначала то, что можно проверить без Firebase, потом то, что требует эмулятора,
и лишь в самом конце — боевые данные.

### Шаг 1 — Чистая логика видимости ✅ сделано

`src/lib/model/visibility.ts` + тесты.

Раскладка свойств по бакетам и склейка при чтении. Здесь возникает утечка приватности, если ошибиться,
и Firestore эту ошибку не поймает: правила стерегут документы, а не то, что мы в них положили.

Сделано:
- `Audience = 'everyone' | CircleId[]`, пустой массив = «никому»;
- **умолчание — скрыть**: свойство без записи в карте видимости уходит в приватный бакет;
- дублирование свойства по кругам (следствие решения В3);
- `mergeBuckets` падает при рассогласовании значений, а не выбирает молча;
- 30 тестов; набор проверен мутациями — четыре реалистичные утечки дают 5, 11, 10 и 1 падение.

### Шаг 2 — Типы документов Firestore 🔲

`src/lib/model/schema.ts`: типы `UserRoot`, `ProfileBucket`, `Circle`, `Friendship`, `AudienceHint`,
`Point`, `DimRating`, `RelationsDoc`. Схема живёт в TypeScript, а не в правилах
(`researches/02`, проблема №3).

Плюс конвертеры `toFirestore` / `fromFirestore` и валидаторы границ (длины строк, диапазоны дат),
покрытые тестами. Firestore-зависимостей здесь нет.

### Шаг 3 — Правила безопасности 🔲

`firestore.rules` по §3.4 предложения. Ключевые:
- `profile/everyone` читает любой подтверждённый — без обращений к базе (дёшево для списка 250 карточек);
- `profile/friends` — через `friendships/{pair}.status == 'accepted'`;
- `profile/{circleId}` — через `exists()` по членству в круге владельца;
- `profile/private` и `users/{uid}` — только владелец;
- круги и их состав — только владелец;
- `points/*` — владелец на запись, вычислитель через Admin SDK на чтение;
- `relations/{uid}` — владелец на чтение, никто из клиентов на запись;
- админ — через custom claim, а не список email.

### Шаг 4 — Тесты правил на эмуляторе 🔲

`@firebase/rules-unit-testing` + эмулятор Firestore (Java уже установлена, см. `STATUS.md`).

Проверять надо **отказы**, а не разрешения. Минимальный набор:
- незнакомец не читает `profile/friends`, `profile/{circle}`, `profile/private`;
- «друг» с `status: 'pending'` не читает `profile/friends` — согласие обязано быть взаимным;
- принять запрос дружбы может только адресат, не отправитель;
- зритель не может прочитать чужой `users/{uid}/groups/*` (состав кругов приватен);
- зритель читает `audience/{он}`, но не `audience/{кто-то другой}`;
- клиент не может писать в `relations/*` и не может выставить себе `dirty`;
- оценка вне 0…10 отвергается; чужая оценка не пишется.

Мутационная проверка обязательна: ослабить правило → тест обязан покраснеть. Иначе это не тест.

### Шаг 5 — План миграции 🔲

Отдельный документ `plans/02_migration.md`. Соответствие коллекций уже расписано в `researches/04` §5.
Принципы: сначала на копии боевой базы, идемпотентный скрипт, сверка числом документов и контрольными
суммами, обратимость (старые коллекции не удаляются до подтверждения), пересчёт связей обязан дать те
же числа (паритет ядра проверен тестами).

Видимость при миграции: всё → `everyone` (решение владельца, В5). Круги и дружба пусты.

## Что уже проверено фактами

| Факт | Следствие | Источник |
|------|-----------|----------|
| Firestore читает документ целиком, поля скрыть нельзя | профиль разбит на бакеты | [док][fields] |
| Бэкенд без входящих портов | сервера-фильтра нет, сторож — только правила | интервью №001, В3 |
| Правила не перебирают значения map | ось = отдельный документ | `researches/04`, Ф3 |
| Круги приватны, их id знает только владелец | нужен документ-подсказка `audience/{viewerUid}` | интервью №002 |
| Дружба симметрична | живёт вне кругов, в `friendships/{pair}` | интервью №002, В1 |

[fields]: https://firebase.google.com/docs/firestore/security/rules-fields

## Риски

- **Утечка через раскладку.** Правила её не ловят. Закрыто чистым модулем + мутационными тестами (шаг 1).
- **Рассогласование дублей.** Свойство в нескольких бакетах должно совпадать. `mergeBuckets` падает,
  а не молчит; запись — одним `WriteBatch`.
- **Стоимость чтений.** Просмотр профиля = `2 + k` чтений плюс по одному `get()`/`exists()` на бакет.
  Список карточек — 250 дешёвых чтений. Замерить на эмуляторе до боевого запуска.
- **Удаление круга** затрагивает бакет профиля и все `audience/*`. Многодокументная операция; если не
  влезает в батч — задача вычислителя.
