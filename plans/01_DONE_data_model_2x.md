## Порядок работ

Снизу вверх: сначала то, что можно проверить без Firebase, потом то, что требует эмулятора,
и лишь в самом конце — боевые данные.

### Шаг 1 — Чистая логика видимости ✅ сделано

`src/lib/model/visibility.ts` + тесты.

Раскладка свойств по бакетам и склейка при чтении. Здесь возникает утечка приватности, если ошибиться,
и Firestore эту ошибку не поймает: правила стерегут документы, а не то, что мы в них положили.

Сделано:
- `Audience = 'everyone' | CircleId[]`, пустой массив = «никому»;
- **умолчание — скрыть**: свойство без записи в карте видимости уходит в приватный бакет;
- дублирование свойства по кругам (следствие решения В3);
- `mergeBuckets` падает при рассогласовании значений, а не выбирает молча;
- 30 тестов; набор проверен мутациями — четыре реалистичные утечки дают 5, 11, 10 и 1 падение.

### Шаг 2 — Типы документов Firestore ✅ сделано

`src/lib/model/schema.ts` + тесты. Схема живёт в TypeScript, а не в правилах
(`researches/02`, проблема №3).

Сделано:
- документы: `UserRootDoc`, `ProfileBucketDoc`, `GroupDoc`, `GroupMemberDoc`, `AudienceHintDoc`,
  `FriendshipDoc`, `PointDoc`, `DimRatingDoc`, `RelationsDoc`, `DimDoc`, `SuggestionDoc`;
- **границы значений взяты из 1.x без изменений** — иначе миграция отвергла бы данные, которые люди
  уже заполнили. Отдельный тест прибивает константы гвоздями: параметризованные тесты не заметили бы
  тихого ужесточения;
- **логика дружбы** как источник истины для правил: принять запрос может только адресат, только из
  статуса `pending`, и никогда — отправитель;
- `isRealDate` отличает 31 февраля от настоящей даты, но **хранение этого не запрещает**: в 1.x
  проверялись только диапазоны, и такие записи могли попасть в базу;
- два умолчания видимости, которые легко перепутать: новый пользователь — **всё скрыто**;
  мигрируемый — **всё видно всем**, потому что таким оно у него и было.

29 тестов; набор проверен мутациями — шесть реалистичных поломок (дружба без согласия, посторонний
принимает запрос, новый пользователь публичен, дробная оценка, тихое ужесточение границы, повторное
принятие) дают падения.

Не сделано намеренно: конвертеры `toFirestore` / `fromFirestore`. Они появятся вместе с адаптером
Firestore, когда будет что адаптировать; отметки времени пока `Millis`.

### Шаг 2.5 — Терминология: круг → группа ✅

Владелец говорит «группы», коллекция называется `groups`. Тип в коде назывался `CircleId`.
Переименовано в `GroupId` / `MAX_CUSTOM_GROUPS` / `Viewer.groups`: одно каноническое написание
(`AGENT_GUIDE.md` → «Идентичность проекта»). В прозе «круг» остаётся как пояснение сути —
это приватный список владельца, а не общая сущность.

### Шаг 3 — Правила безопасности ✅ сделано

`firestore.rules`. Правила проверяют **права доступа и границы**; форму данных знает `schema.ts`.
Условие дружбы повторяет `canAcceptFriendship()`, а не изобретает заново.

Сделано:
- `profile/everyone` читает любой подтверждённый — **без обращений к базе**: это самое частое чтение
  в продукте (250 карточек связей), и оно обязано быть дешёвым;
- `profile/friends` — через `friendships/{pair}.status == 'accepted'`; статус `pending` доступа не даёт;
- `profile/{groupId}` — через `exists()` по членству в группе владельца; идентификаторы `everyone`,
  `private` и `friends` явно исключены, чтобы правило нельзя было обойти группой с таким именем;
- `users/{uid}`, `profile/private`, группы и их состав — только владелец;
- `friendships` — создать может участник от своего имени, принять **только адресат** из `pending`,
  и он не может переписать, кто отправлял запрос;
- `points/*` — оценки видит только владелец; клиент может выставить `dirty`, но не снять (это делает
  вычислитель); значение оси — целое 0…10, лишние поля отвергаются;
- `relations/{uid}` — владелец читает, **клиенту запись запрещена полностью**;
- админ — через custom claim, а не список email в тексте правил;
- умолчание `match /{document=**} { allow read, write: if false; }`.

### Шаг 4 — Тесты правил на эмуляторе ✅ сделано

`tests/rules/rules.test.ts`, 56 тестов. Запуск: `npm run test:rules`.
`@firebase/rules-unit-testing` + эмулятор Firestore на порту **8181** (8080 занят `llama-swap`);
адрес берётся из `FIRESTORE_EMULATOR_HOST`, поэтому тест не знает про порт.

Проверяются **отказы**, а не разрешения: сломанное разрешение видно сразу, сломанный отказ — никогда.
Среди прочего:
- запрос дружбы в статусе `pending` не открывает `profile/friends`;
- отправитель не может принять собственный запрос; посторонний — чужой;
- группа с именем `private` или `friends` не подменяет проверку;
- участник группы **не знает**, в какой он группе, и не видит её состав;
- документ-подсказка `audience/{я}` читается, `audience/{другой}` — нет; и сама по себе она **не даёт
  прав**: без членства бакет всё равно закрыт;
- подтверждённый друг **не видит оценок по осям** — только общий результат;
- владелец не может переписать свои связи;
- автор не читает даже свою заявку на ось.

**Мутационная проверка пройдена.** Восемь ослаблений правил:
`pending`≈`accepted`, отправитель принимает свой запрос, приватный бакет открывается группой,
владелец пишет связи, чужая подсказка читается всеми, чужие оценки видны, оценка вне диапазона,
и снятие умолчания «запрещено». Дают 1, 1, 1, 1, 1, 1, 1 и **39** падений соответственно.

### Шаг 5 — План миграции ✅ сделано

`plans/02_migration.md`. Ключевая находка при написании: в 1.x документы связаны с пользователем **по
полю `owner_uid`, а не по ключу** (`ndimids`/`relations` имеют собственные id). Значит группировать
надо по `owner_uid`, и первым делом проверить инвариант «один uid → один ndimids»; нарушители
останавливают миграцию и зовут человека.

Порядок: доступ+копия (человек) → инвентаризация → чистые функции трансформации с тестами →
прогон на песочнице → сверка суммой оценок и пересчётом связей → боевой go/no-go → уборка (недели
спустя). Старое не удаляется, пока 2.0 не докажет, что работает.

Сам скрипт миграции **не пишется сейчас**: мигрировать некуда, пока нет фронтенда. Чистые функции
трансформации можно написать в любой момент (боевого доступа не требуют), прогон — когда 2.0 готов
принять людей.

## Что уже проверено фактами

| Факт | Следствие | Источник |
|------|-----------|----------|
| Firestore читает документ целиком, поля скрыть нельзя | профиль разбит на бакеты | [док][fields] |
| Бэкенд без входящих портов | сервера-фильтра нет, сторож — только правила | интервью №001, В3 |
| Правила не перебирают значения map | ось = отдельный документ | `researches/04`, Ф3 |
| Круги приватны, их id знает только владелец | нужен документ-подсказка `audience/{viewerUid}` | интервью №002 |
| Дружба симметрична | живёт вне кругов, в `friendships/{pair}` | интервью №002, В1 |

[fields]: https://firebase.google.com/docs/firestore/security/rules-fields

## Риски

- **Утечка через раскладку.** Правила её не ловят. Закрыто чистым модулем + мутационными тестами (шаг 1).
- **Рассогласование дублей.** Свойство в нескольких бакетах должно совпадать. `mergeBuckets` падает,
  а не молчит; запись — одним `WriteBatch`.
- **Стоимость чтений.** Просмотр профиля = `2 + k` чтений плюс по одному `get()`/`exists()` на бакет.
  Список карточек — 250 дешёвых чтений. Замерить на эмуляторе до боевого запуска.
- **Удаление круга** затрагивает бакет профиля и все `audience/*`. Многодокументная операция; если не
  влезает в батч — задача вычислителя.
