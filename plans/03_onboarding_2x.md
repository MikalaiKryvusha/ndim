# План 03 — Онбординг 2.0 без трения: демо → гость → аккаунт

> Реализация решения боли `ideas/09` (трение онбординга 1.x). Исследование —
> `researches/10_onboarding_friction.md`, решения владельца —
> `interviews/interview_004_onboarding_strategy.md` (закрыто 2026-07-11):
> **В1 = A** оба слоя (демо `ideas/10` + гость `ideas/11`) · **В2 = A+** Google + волшебная
> ссылка + известные OAuth-сервисы, без паролей · **В3 = A** гость невидим другим ·

## Целевой путь пользователя (сквозной)

```
Лендинг ──▶ Демо похожести (без аккаунта, 30 секунд, ядро в браузере)
              │  «Создай своё пространство»
              ▼
        Гостевой режим (anonymous auth: свои измерения, оценки, своё пространство;
              │   никому не виден; труд копится в Firestore под гостевым UID)
              │  логичный момент: «сохранить результаты» / «стать видимым» / «дружба»
              ▼
        Аккаунт (Google / волшебная ссылка / OAuth-сервисы; linkWithCredential:
                 UID тот же, ВЕСЬ труд гостя сохраняется)
```

Каждый этап воронки пишет счётчик (В4) — увидим, где рвётся.

## Порядок работ

Снизу вверх: сначала то, что проверяется без браузера, затем стенд, в конце — живой домен.

### Этап 2 — Гостевой режим (`ideas/11`, часть 1) ✅ на стенде (2026-07-11; прод — с публикацией 2.0)

- ✅ **Anonymous Auth на стенде**: `signInGuest()` + `ensureSpaceExists()` в

  `dirtyStamp()` честно помечает точку гостя `guest: true` (иначе правила отвергнут).
  Прод-включение анонимного провайдера — с публикацией 2.0 (фаза 5).
- ✅ **Правила**: `isGuest()` по `sign_in_provider == 'anonymous'` + честный флаг `guest` на
  точке (`honestGuestFlag`: гость не скроет, полноценный не прикинется). Разрешено: свои
  `points/`+оценки, каталог измерений, свои relations, приватные бакеты. Запрещено: публикация
  бакетов everyone/friends, группы, `audience`, дружба, заявки на измерения (В3 = A). 17 тестов
  отказов (73 всего), 3 мутации правил пойманы.
- ✅ **Вычислитель**: точки с `guest: true` не попадают ни в чей топ (даже другого гостя),
  но сам гость получает свой топ против публичных точек. Тест `npm run test:calc` (5 проверок
  на эмуляторе), мутация «фильтр снят» ловится. Точка входа теперь срабатывает только при
  прямом запуске — модуль импортируем тестами.
- ✅ **Автоочистка сирот**: `cleanupStaleGuests()` в начале каждого цикла вычислителя —
  guest-точки с `lastSync` старше 30 дней (и не dirty) удаляются целиком (точка, оценки,
  relations, users-дерево). Труд полноценных людей неприкосновенен при любом возрасте —
  это охраняется тестом (мутация «фильтр гостей снят» ловится). `test:calc` — 9 проверок.
- ✅ **UI-флоу гостя утверждён** (2026-07-11, `design/guest-flow-mockups.html`): база —

  Метафора: гость = пунктирный контур, аккаунт = сплошной. Правки уже внесены в макет.
- ✅ **UI гостя реализован по утверждённому V1**: пунктирная пилюля «◌ гость» в AppBar,

### Этап 3 — Аккаунт без пароля (`ideas/11`, часть 2) ✅ на стенде (2026-07-12)

- ✅ **Макет V4 «Врезка» утверждён** владельцем с правками текстов; правила текста продукта
  закреплены в шапке `design/login-mockups.html`, словарь — в `AGENT_GUIDE.md`.
- ✅ **`src/lib/data/account.ts`**: `linkGoogle()` (linkWithPopup) и почтовая ссылка
  (`sendLoginLink` → `completeLoginLink` через `linkWithCredential`). Строго `link*`, не `signIn*` —
  UID сохраняется, труд гостя остаётся.
- ✅ **Расклейка точки — обязательная часть апгрейда** (`promoteGuestPoint`): пока человек был
  гостем, его точка помечена `guest: true`. Не снять флаг = человек с настоящим аккаунтом остаётся
  невидимым для всех, а автоочистка удалит его данные через 30 дней. Плюс `getIdToken(true)`:
  правила смотрят на клейм `sign_in_provider`, а не на клиентский `isAnonymous`.
- ✅ **UI по V4**: карточка гостя разворачивается на месте — факты → выбор способа → «Мы отправили
  Вам письмо» → «Профиль сохранён». Честные тексты ошибок (`already-in-use`, `expired-link`, …).
- ✅ **Сквозной сценарий на стенде** (Playwright + эмуляторы, письмо забирается из oobCodes
  эмулятора Auth): гость оценил измерение → аккаунт по ссылке из письма → **UID тот же**, почта
  привязана, оценки на месте, `guest: false`, точка грязная. Проверено **мутацией**: «флаг не
  снят» → 2 падения (и правило `honestGuestFlag` само отвергает такую запись).
- 🔲 Осталось: Google-вход проверен только кодом (эмулятор требует интерактивного окна);
  прод-включение провайдеров — с публикацией 2.0.

#### Решения владельца (2026-07-12) — оба вопроса закрыты

- ✅ **Почта или Google уже принадлежат другому профилю** (`already-in-use`): оставляем как есть —
  честно говорим «Этот способ входа уже связан с другим профилем. Чтобы сохранить текущие
  результаты, создайте аккаунт на другую почту». **Слить два профиля молча нельзя: чей-то труд
  пришлось бы выбросить.** Автоматического входа в старый профиль не делаем.
- ✅ **Проверка Google-входа руками — отложена** до включения провайдеров на проде (фаза 5).
  Тогда агент **обязан попросить владельца** проверить вход через Google живьём: эмулятор требует
  интерактивного окна, автотест его не проходит. Пункт вынесен в чеклист фазы 5 в `STATUS.md`.

### Этап 4 — Воронка (В4) ✅ на стенде (2026-07-12)

- ✅ **`space/funnel/days/{дата}`** — четыре счётчика на сутки: `landing_view`, `demo_touch`,
  `guest_start`, `account_created`. Больше в документе нет ничего: ни UID, ни почты, ни
  устройства. Восстановить человека по этим числам нельзя даже теоретически.
- ✅ **Правила — самое тесное место этапа**: писать может кто угодно (первый шаг случается до
  входа), поэтому разрешены только эти четыре имени полей, за одну запись меняется **ровно один**
  счётчик и **ровно на +1**, остальные обязаны остаться нетронутыми; уменьшать нельзя; читает
  только админ. **8 новых тестов отказов** (81 в `test:rules`), **2 мутации пойманы**:
  «любой рост вместо +1» и «открытый набор полей» — каждая роняет ровно свой тест.
- ✅ **`src/lib/data/funnel.ts`**: `track(step)` — не бросает, не тормозит, один визит = один шаг
  каждого вида (`sessionStorage`). Firebase подгружается **динамическим импортом**: иначе SDK
  затащило бы в главный бандл лендинга, а он обязан оставаться лёгким (Lighthouse 100).
  Проверено: `firebase` в чанках лендинга отсутствует.
- ✅ **Сквозной прогон на стенде**: лендинг → два клика по звёздам → гость → аккаунт по ссылке из
  письма. Воронка: `1 / 1 / 1 / 1` — то есть **два клика дали ОДИН шаг**: считаем людей, а не клики.
- 🔲 На публичном домене воронка — осознанный no-op: боевого конфига Firebase ещё нет
  (`src/lib/firebase.ts` ходит только в эмуляторы). Включится вместе с публикацией 2.0 (фаза 5).
- 🔲 Показ воронки владельцу — в экране «Пространство» (`ideas/06`, макеты ждут утверждения).
- 🔲 App Check против накрутки счётчиков — отдельной задачей перед публичным запуском.

## Критерий готовности плана

Человек с рекламы проходит путь «лендинг → потрогал → гость → аккаунт» без единого пароля и
без потери введённого, каждый переход измерим, ни одно правило приватности не ослаблено
(мутационные тесты правил зелёные).
