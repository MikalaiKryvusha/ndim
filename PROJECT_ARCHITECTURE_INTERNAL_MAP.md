## Ключевые абстракции

| Абстракция | Что это *по сути* | За что отвечает |
|------------|-------------------|-----------------|
| **Измерение** (`dim`) | Ось человеческого качества, предложенная сообществом: «люблю тишину», «рано встаю». Не категория, а **координатная ось**. | Задаёт одну размерность пространства. Живёт в коллекции `dims`, имеет рейтинг (`stars`/`rates`). |
| **Оценка** (`stars`) | Целое число 0…10 — где человек стоит на этой оси. | Одна координата человека. |
| **Пространство пользователя** (`user_dims`) | Отображение `dim_id → stars` — набор осей, которые человек про себя заполнил. | Человек = **точка** в своём собственном подпространстве. У разных людей подпространства разные. |
| **NDim-ID** (`ndimid`) | Профиль-точка: `owner_uid`, `owner_email`, `user_dims`, `sync_status`, `time{created, last_sync, updated, last_sign_in}`. | Единица, между которой считаются связи. |
| **Общее пространство** | Пересечение осей двух людей: `K = dims(A) ∩ dims(B)`. | Единственное место, где двоих вообще можно сравнивать. |
| **Связь** (`relation`) | Посчитанная пара «владелец → гость» с метриками близости. | Результат работы ядра. Хранится в `relations`, читать может только владелец. |
| **Похожесть** (`similarity`) | Итоговое число 0…100 — насколько двое похожи. | То, ради чего существует продукт. |
| **Метрики пространства** (`space`) | Агрегаты по всей платформе: диаметр общего пространства, счётчики. | Публичная витрина (`space/public_metrics`) и админские (`space/metrics`). |
| **Предложение** (`suggestion`) | Заявка пользователя на новую ось измерения (5…300 символов). | Как пространство растёт: оси предлагают сами люди. |

## Математическое ядро — как считается похожесть

Единственный по-настоящему ценный код проекта: функция `calculateRelation(owner, guest)`.
Исходник дословно — `researches/03_similarity_core_1x_source.md`.

Даны два человека, каждый — точка в своём наборе осей. Считаем **только по общим осям**:

```
K = dims(owner) ∩ dims(guest)                          общие оси
d = √( Σ_{k∈K} (owner[k] − guest[k])² )                евклидово расстояние в общем пространстве
D = √|K| · 10                                          диаметр: диагональ |K|-мерного куба со стороной 10

близость     proximity   = 1 − (d / D)^0.7             ∈ [0,1]   — насколько близко стоят
общность     commonality = 2|K| / (|dims(owner)| + |dims(guest)|)  ∈ [0,1]   — коэффициент Дайса
ПОХОЖЕСТЬ    similarity  = proximity × commonality     ∈ [0,1]   → округляется до целых процентов
```

**Почему именно так — смысл, а не формула:**

- **Расстояние нормируется диаметром**, а не числом осей. Иначе люди, заполнившие много общих осей,
  автоматически считались бы дальше друг от друга, чем люди с одной общей осью.
- **Показатель `^0.7`** (вместо линейного `1 − d/D`) поднимает чувствительность в области малых
  расстояний: система лучше различает «очень близко» от «просто близко» — а именно там живёт ответ,
  который нужен пользователю. В коде это прямо помечено как замена старой линейной формулы.
- **Общность по Дайсу** наказывает за узость пересечения. Двое, совпавшие идеально по одной общей оси
  из пятидесяти, — не похожие люди, а случайное совпадение. Дайс делает `similarity` малой.
- **Произведение, а не сумма.** Похожесть требует *и* близости, *и* широкого пересечения. Ноль в любом
  множителе обнуляет результат. Это осознанный отказ от компромиссных «средних».

**Крайние случаи (проверить тестами в фазе 0):**
- `|K| = 0` → общего пространства нет, `relation_data = null`, связь не создаётся.
- `owner = guest` (человек сам с собой) → `d = 0` → `proximity = 1`, `commonality = 1` → `similarity = 100`.
  Используется намеренно: так считаются метрики собственного пространства пользователя.
- `d = D` (максимально далеко в общем пространстве) → `proximity = 0` → `similarity = 0`.

**Производные метрики.** Ядро возвращает ещё ~20 величин: размеры и диаметры пространств владельца,
гостя и общего, их взаимные отношения. ⚠️ Все они сохраняются под обфусцированными именами `a…t`
через «матрицу соответствия». **Это долг, а не фича:** в 2.0 метрикам вернуть настоящие имена.

## Как абстракции взаимодействуют

Поток в версии 1.x — и место, где всё ломается:

```
человек ──заполняет оси──▶ ndimids/{id}.user_dims ──┐
сообщество ──предлагает──▶ suggestions ──▶ dims ────┤
                                                     │
                     ┌───────────────────────────────┘
                     ▼
      ⚠ БРАУЗЕР АДМИНА (Chrome на телефоне Николая, всегда открыт)
         importAdminFunctions() тянет приватный ES-модуль
         /admin/adminFunctions.js из Firebase Storage
                     │
                     ▼
         calculateRelation(owner, guest) для КАЖДОЙ пары   ← O(N²) в UI-потоке
                     │
                     ▼
         топ-250 самых похожих на каждого владельца
                     │
                     ▼
              relations/{id}  ──читает только владелец──▶ экран «Мои связи»
```

**Что здесь не так (по `GOAL.md`):**

1. **Вычислитель — это телефон.** Роль сервера исполняет вкладка Chrome. Если телефон уснул,
   разрядился или потерял сеть — платформа перестаёт находить людей.
2. **Приватность через обфускацию.** Ядро прячется в Storage и грузится динамическим `import()`
   только для аккаунтов из списка админов. Это не безопасность, а её имитация.
3. **O(N²) без пагинации.** Каждый с каждым. Не масштабируется дальше нескольких тысяч профилей.
4. **Права зашиты в правила.** Админский доступ определяется буквальным списком email внутри
   `firestore.rules` (`kotkrinik@yandex.ru`, `kotkrinik@gmail.com`).

## Инварианты и законы модели

Эти правила должны выполняться всегда. Их нарушение — баг, даже если код исполняется.

- **Сравнение возможно только в общем пространстве.** Никогда не додумывай отсутствующую координату
  (нулём, средним, чем угодно). Нет общей оси — нет сравнения.
- **`similarity = proximity × commonality`.** Ни одна из компонент не может «компенсировать» другую.
- **Все метрики симметричны по построению**, кроме тех, что явно нормированы «по владельцу»
  (`*_rate_of_owner`). Не путай их местами.
- **Связь принадлежит владельцу.** `relations` читает только `owner_uid`. Похожесть — не публичная
  величина.
- **Оси предлагают люди, а не платформа.** Пространство измерений растёт снизу. Платформа не решает,
  какими качествами описывать человека.
- **Продукт не удерживает.** Найденный человек — это успех, а не потерянная сессия. Любая механика,
  повышающая вовлечённость ценой качества поиска, нарушает цель проекта (`GOAL.md`, `PHILOSOPHY.md` →
  «Карма»).

## Ключевые решения, вшитые в архитектуру

| Решение | Почему так | Цена |
|---------|-----------|------|
| Оси задаёт сообщество, не платформа | Никто, кроме людей, не знает, чем люди отличаются | Пространство разреженное: пересечения малы, Дайс жёстко наказывает |
| Похожесть = близость × общность | Отказ от «средних» компромиссов; оба фактора обязательны | Похожесть падает быстро — но честно |
| `^0.7` вместо линейной близости | Различать «очень близко» от «близко» — там и есть ответ | Магическая константа без обоснования в коде |
| Считать заранее и хранить топ-250 | Мгновенная выдача пользователю | Нужен фоновой вычислитель — которого нет |
| Firebase как вся платформа | Один человек без опыта смог это построить | Хостинг не индексируется, бэкенда нет, домен потерян |

Эти решения — прямые входы в `MASTER_PLAN.md`. Первые три **сохранить** (это и есть идея), последние
два **заменить** (это и есть проблема).
