# Idea 07 — Автоматика билдов и деплоя (версии — по команде владельца)

> Агент причесал исходный текст автора в структурный вид. Смысл и формулировки автора сохранены.

## Суть (слова владельца, структурированные)

В 1.x были настроены хуки в VS Code: каждый коммит автоматически повышал **версию сборки**
приложения, а GitHub Actions запускал **автоматический деплой** нового кода. Нужна такая же
автоматизация для NDim Space 2.0, с ясным правилом для ИИ-агентов:

- **номера билдов** повышаются автоматически, агенты понимают как;
- **номера версий** оформляются **по команде владельца** (не автоматически).

## Предложение агента (обсуждается при реализации)

- **Билд не хранить — вычислять из git.** Вместо хука, который пишет числа в файл на каждом
  коммите (шум в истории + хуки у агентов различаются), билд выводится из самой истории:
  `git rev-list --count HEAD` (монотонный номер) + короткий хеш. Никакой ручной дисциплины,
  нечему сломаться, у любого агента «как повышается билд» = «сделай коммит» (Оккам).
- **Версия** — `package.json`, бампается только по команде владельца («сделай релиз») — скилл
  `/release` уже описывает этот ритуал (бамп + тег + GitHub Release).
- **Инжекция в приложение**: Vite `define` подставляет `версию + билд + дату сборки` в код —
  показывается в UI (см. `ideas/06`). Вычислителю то же самое вшивается при сборке Docker-образа,
  он публикует это в `space/server`.
- **CI/CD (GitHub Actions)**: workflow на push в `main` — `npm test` → `npm run e2e` →
  `npm run kaif:check` → деплой **лендинг-сайта** `ndimspace` через Firebase. Сервисный аккаунт
  `github-action-855638138@ndim-space` уже существует со времён 1.x — проверить права и
  переиспользовать. Деплой вычислителя в GitHub Actions не нужен: сервер — этот ПК,
  перезапуск контейнера локальный (`docker compose up -d --build`).
- **Границы безопасности**: деплой из CI касается только сайта `ndimspace` (лендинг+экраны);
  живой 1.x (`ndim-space.web.app`) из CI недосягаем. `kaif:check` в CI — обязательный шаг.
