## 1. Общая картина: «сервером» был телефон владельца

В 1.x не было ни Docker, ни Admin SDK. «Сервер синхронизации» — это **админ-страница самого
приложения**, открытая в браузере на устройстве владельца (обычно телефон). Владелец нажимал
кнопку на `synchronizer_console` — и страница становилась демоном:

- `runAutomaticSynchronizerWorker()` запускал `runHourlyAutomaticSynchronizerWorker_v2()`;
- воркер планировал сам себя через `setTimeout` на **начало следующего часа** (`app.js:10358–10369`);
- права — обычный Firebase Auth владельца; правила пускали админа по email (антипаттерн, в 2.0
  заменён на custom claim).

Отсюда главное следствие: **все расчёты были бесплатны (браузер), а платны только запросы к
Firestore** — и вся архитектура выстроена вокруг минимизации именно запросов.

## 2. Модель данных, на которой держалась экономия

Три кита — все три «один документ = целая сущность»:

| Документ | Что внутри | Экономия |
|---|---|---|
| `ndimids/{uid}` | **ВСЕ оценки человека** одной картой `user_dims: {dim_id: value}` + `sync_status` + `time.{updated, last_sync, last_sign_in}` | 1 чтение = весь человек. В 2.0 оценки — подколлекция `points/{uid}/dims/`: человек стоит 1 + N чтений |
| `relations/{uid}` | **Весь топ-250** одной JSON-строкой в поле `relations` (поля связи обфусцированы `a…t`, см. `researches/03`) | 1 чтение = весь топ; 1 запись = весь топ |
| `dims/dims_list` | **Весь каталог измерений** одной JSON-строкой, включая рейтинги (`stars`, `rates`, `rating`) | 1 чтение = каталог из 5111 измерений; 1 запись = обновление рейтингов всех измерений разом |

Плюс статусная машина на `sync_status`:

- `never` — человек зарегистрировался, но ни разу не менял свой NDim ID. **Такие не загружаются
  в кэш и не синхронизируются никогда** — для них нечего считать.
- `unsynchronized` — человек менял оценки после последней синхронизации. Клиент ставит этот
  статус при каждом добавлении/изменении/удалении оценки (`app.js:8845, 8891, 8957`).
- `synchronized` — синхронизатор пересчитал топ и снял флаг (`markNdimidAsSynchronized`).

Это прямой предок флага `dirty` в 2.0 (`researches/02`, проблема №9).

## 3. Кэш в памяти — сердце экономии

`ndimids_cache` — обычный объект в памяти страницы (`app.js:274`), живёт, пока открыта вкладка:

1. **Первый цикл** (`first_sync`): один запрос `where sync_status in [unsynchronized, synchronized]`
   — все, кто когда-либо менял свой NDim ID (**без** `never`!). Полная загрузка кэша.
2. **Каждый следующий час** (`hourly_sync`): запрос **только** `where sync_status in [unsynchronized]`
   — дельта. Полученные документы перезаписывают свои записи в кэше.
3. **Раз в сутки** (`daily_sync`): запрос `(last_sign_in > сутки назад && last_sync < сутки назад)
   OR unsynchronized` — дополнительно освежает в кэше время логина активных людей.

Все пересчёты идут **против кэша, не против базы**: топ владельца считается циклом по
`ndimids_cache` без единого чтения из Firestore (`app.js:10139`).

## 4. Типы синхронизаций: кого пересчитывать

`syncAndUpdateRelations(sync_type)` собирает список `ndimids_need_to_be_synced`:

| Тип | Когда | Кто пересчитывается | Что пишется |
|---|---|---|---|
| `first_sync` | первый цикл после запуска | все `unsynchronized` из кэша | по 1 `relations/{uid}` + 1 статус на каждого |
| `hourly_sync` | каждый час | **только `unsynchronized`** (менявшие оценки за час) | то же — только им |
| `daily_sync` | в 00:00 по Лондону | `unsynchronized` **плюс** `synchronized`, у кого (логин за последние 7 дней) && (не синхронизирован больше суток) | то же — каждому из списка |
| `one_shot` | вручную, кнопкой | **все** из кэша | всем |
| пустой цикл | нечего синхронизировать | никто | **ничего** — один принт «skipped» в консоль |

Каждому из списка: топ-250 против всего кэша (сортировка по похожести `i`, срез 250,
`JSON.stringify`) → **одна** запись `relations/{uid}` → **одна** запись статуса в `ndimids/{uid}`
→ обновление записи в локальном кэше. Детали окна дневной выборки: сдвиг в 59 минут
(`app.js:10081`), чтобы вчерашние синхронизации не выпадали из завтрашнего окна.

## 5. Контракт свежести, который принял автор

Ключевое продуктовое решение 1.x, на котором держится вся экономия записей:

- **Свой топ** обновляется в течение **часа** после изменения оценок.
- **Твоё появление в чужих топах** — лениво: у **активных** людей (логин за 7 дней) топы
  освежаются **раз в сутки** ночью; у неактивных — **протухают до их возвращения**
  (вернулся → `last_sign_in` обновился → этой же ночью его топ пересчитан).

То есть 1.x **никогда не пересчитывал всех при каждом изменении** — изменение расползалось
по чужим топам за сутки, и только по активным. Это осознанная цена, а не недосмотр.

## 6. Рейтинги измерений и метрики — попутным грузом

- После каждого цикла с реальной работой: `getAllDimsList()` (1 чтение),
  `calculateRatings(all_dims_list, ndimids_cache)` — рейтинги всех измерений из кэша,
  `updateDimsList()` (1 запись). Итого 1+1 запрос на весь каталог.
- Метрики пространства (`space/metrics`) пишутся **только после реальных синхронизаций**
  (и на старте воркера) — не каждый цикл. Пустой час не пишет в базу вообще ничего.

## 7. Арифметика запросов (почему 1.x был дёшев)

При ~330 людях и ~20 оценках у человека:

| Событие | 1.x | 2.0 до `ideas/14` |
|---|---|---|
| Час без изменений | 1 чтение (пустой запрос) | ~30 холостых циклов: 60+ чтений, 60+ записей |
| Один человек изменил оценки | 1 чтение (его документ) + 2 записи (его топ, его статус) + 2 на dims_list; чужие топы — ночью | полное чтение всех точек (~7000 чтений) + запись **всех** топов (~330 записей) в ближайшие 60 с |
| Сутки жизни Пространства | ~24 запроса + ночной пересчёт активных | ~1440 циклов × (2 запроса + 2 записи) минимум |

## 8. Что из этого уже живёт в 2.0, а что нарушено

Уже унаследовано: `dims/dims_list` (индекс каталога, пересборка только при расхождении размера),
топ-250, флаг `dirty` с выборкой по индексу, счёт каталога агрегацией, `space/public_metrics`.

Нарушено (предмет `ideas/14`): нет кэша между циклами (полное перечитывание), пересчёт и
**перезапись всех** при любой грязной точке, цикл 60 секунд с небесплатной холостой работой
(запрос гостей + чтение stats + 2 записи каждую минуту), запись без сравнения с предыдущим.

## 9. Что переносить НЕ надо (антипаттерны 1.x — уже решено в 2.0)

- `owner_email` внутри `relations` — ПДн в чужом документе (`researches/03`, поля `e`/`f`).
- Обфускация полей `a…t` — в 2.0 поля именованы честно.
- Права админа списком email в правилах — в 2.0 custom claim.
- Сервер в браузере на телефоне — в 2.0 Docker-служба с Admin SDK (интервью №001, В3).
- Оценки картой в одном документе — в 2.0 осознанно подколлекция (правила стерегут каждую
  оценку; смена модели — отдельное архитектурное решение владельца, если когда-то понадобится).
