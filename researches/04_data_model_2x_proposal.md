# Исследование 04 — Модель данных NDim Space 2.0: предложение

> **Что это.** Проект новой модели данных Firestore. Владелец разрешил её спроектировать заново
> (`researches/02`), задал два требования — **настраиваемая видимость по каждому свойству** и
> **уважительная миграция** — и снял с повестки историю оценок.
>
> Документ ведёт от **проверенных технических фактов** к устройству модели. Развилки, которые агент не
> имеет права закрыть один, вынесены в `interviews/interview_002_visibility_friends_groups.md`.
>
> **Живой справочник — не помечается DONE.** Дата: 2026-07-10.

## 1. Требования владельца

| № | Требование | Источник |
|---|-----------|----------|
| Т1 | Видимость личных данных настраивается **по каждому свойству в отдельности** | чат, 2026-07-10 |
| Т2 | Аудитория свойства — **набор кругов владельца, объединённых ИЛИ**; «Друзья» — дефолтный круг с взаимным согласием; своих кругов до 10 | интервью №002, В1–В3 |
| Т3 | **Историю оценок не делаем** — «сложно, пока не делаем» | чат, 2026-07-10 |
| Т4 | Миграция существующих данных — **уважительная**: без потерь, обратимая, сначала на копии | `researches/02` |
| Т5 | Бэкенд — Docker на домашнем ПК, **только исходящие соединения** | интервью №001, В3 |
| Т6 | Настройка видимости касается **только профиля**; оценки по осям не видит никто, кроме владельца и вычислителя | интервью №002, В4 |
| Т7 | При миграции всё остаётся **видимым всем**, как в 1.x | интервью №002, В5 |

Т2 вводит в продукт сущности, которых в 1.x не было: **круги (группы)** и **дружбу**.

## 2. Проверенные факты, которые определяют решение

### Ф1. Firestore не умеет скрывать отдельные поля при чтении

Официальная документация, [Control access to specific fields][fields]:

> **«Reads in Cloud Firestore are performed at the document level. You either retrieve the full
> document, or you retrieve nothing.»**

Правила безопасности управляют доступом **к документу целиком**. Рекомендация Google — та же самая:
*«put them in a separate document»*, вплоть до подколлекций
(`/employees/{id}` публично, `/employees/{id}/private/finances` — нет).

На **запись** контроль по полям есть: `hasOnly()`, `hasAll()`, `diff().affectedKeys()`, проверка типов.
Но частичная запись невозможна: правило принимает или отвергает всю операцию.

### Ф2. У нас нет сервера, который мог бы фильтровать поля на лету

Владелец выбрал бэкенд-контейнер на домашнем ПК **без входящих портов** (Т5). Значит клиент
**читает Firestore напрямую**, и единственный сторож между ним и данными — правила безопасности.
Вариант «читать через API, который отдаёт только разрешённые поля» отпадает вместе с сервером.

### Ф3. Правила Firestore не умеют перебирать значения map

В rules у `Map` есть `keys()`, `values()`, `size()`, `diff()`, но **нет итерации и предиката «для
всех»**. Проверить «все значения в `dims` лежат в 0…10» одним правилом нельзя.

Отсюда растёт странность 1.x: `user_dims.diff(...).affectedKeys().size() <= 1` — «меняй ровно одну ось
за раз», потому что тогда можно проверить единственный изменённый ключ. Это не каприз автора, а
обход ограничения платформы.

### Вывод

**Т1 + Ф1 + Ф2 ⇒ профиль обязан быть разбит на документы по аудиториям.** Это не вопрос вкуса, а
единственный способ выполнить требование владельца на выбранной архитектуре.

## 3. Структура (после ответов интервью №002)

### 3.0. Модель аудитории

Владелец решил (интервью №002): **группа — это круг владельца**, приватный список людей, который он
собирает сам, односторонне. Названия и состав знает только он. «Друзья» — дефолтная группа с одним
отличием: в неё попадают **по взаимному согласию**. Пользовательских групп — до **10**.

Видимость свойства = **набор групп, объединённых ИЛИ**. Отмечены три группы — люди из этих трёх видят
свойство, остальные нет.

```ts
type GroupId = 'friends' | string;          // 'friends' — дефолтная; остальные — id кругов владельца
type Audience = 'everyone' | GroupId[];     // [] означает «никому»
```

### 3.1. Профиль: документ на группу

```
users/{uid}                              # корень. Читает ТОЛЬКО владелец.
  visibility: { name: Audience, born: Audience, about: Audience, avatar: Audience, gender: Audience }
  settings:   { language: 'ru' | 'en' }
  time:       { created, updated, lastSignIn }

users/{uid}/profile/everyone             # свойства, открытые всем
users/{uid}/profile/friends              # открытые дефолтной группе «Друзья»
users/{uid}/profile/{groupId}            # открытые конкретному кругу владельца
users/{uid}/profile/private              # «никому» — читает только владелец
```

Читатель забирает бакеты, к которым допущен, и **склеивает** их.

> ⚠️ **Дублирование неизбежно.** Свойство с видимостью `['friends', 'g3']` лежит **в обоих**
> документах: зритель может состоять только в одной из этих групп и обязан увидеть свойство. Это прямая
> цена решения В3 (набор аудиторий вместо одной). Стоимость ограничена: свойство копируется максимум в
> 11 документов. Изменение значения свойства — один `WriteBatch` по его бакетам.

`profile/everyone` естественно становится **карточкой** для списков: экран «мои связи» показывает до
250 человек, и ему хватает одного чтения на человека, без обращений к правам.

`email` в профиль **не входит** вообще: это идентификатор, а не свойство (в 1.x он дублировался в трёх
коллекциях — `researches/02`, проблема №5).

### 3.2. Круги владельца и дружба

Круги приватны — их не видит никто, кроме владельца:

```
users/{uid}/groups/{groupId}                    # { name, memberCount, created } — ТОЛЬКО владелец
users/{uid}/groups/{groupId}/members/{memberUid} # ТОЛЬКО владелец
```

Дружба **симметрична**, поэтому не может жить внутри приватного круга одной стороны. Отдельная
коллекция верхнего уровня, ключ — пара uid в лексикографическом порядке:

```
friendships/{minUid}_{maxUid}
  a, b            # uid участников, a < b
  requestedBy     # кто отправил запрос
  status          # 'pending' | 'accepted'
```

Создать может любой из двоих со статусом `pending`; перевести в `accepted` — **только адресат**.
Так «взаимное согласие» из В1 обеспечивается правилами, без участия бэкенда (которого у нас, напомню,
нет для входящих запросов).

### 3.3. Как зритель узнаёт, что ему доступно

Идентификаторы кругов принадлежат владельцу, и зритель их не знает. Перебирать наугад нельзя.
Поэтому владелец ведёт **документ-подсказку** для каждого, кого положил хотя бы в один круг:

```
users/{uid}/audience/{viewerUid}
  buckets: ['friends', 'g3']      # какие бакеты профиля доступны этому зрителю
```

Читать его может **только сам зритель** (и владелец). Названия кругов и другие участники в него не
попадают — только непрозрачные идентификаторы.

> **Подсказка — не источник прав.** Права проверяются правилами независимо, через `exists()` по
> членству. Подсказка лишь отвечает на вопрос «какие документы вообще запрашивать», чтобы клиент не
> ловил отказы. Подделать её нельзя: она пишется владельцем и не даёт доступа сама по себе.

Порядок чтения чужого профиля:
1. `profile/everyone` — всегда (1 чтение);
2. `audience/{я}` — если существует (1 чтение);
3. перечисленные там бакеты (`k` чтений).

Итого `2 + k`. Для списка из 250 карточек — 250 чтений, без правил с обращениями к базе.

### 3.4. Правила чтения

```javascript
function verified() {
  return request.auth != null && request.auth.token.email_verified == true;
}
function isOwner(ownerId) { return request.auth != null && request.auth.uid == ownerId; }

// friendships/{pairId}: pairId == min(a,b) + '_' + max(a,b)
function friendshipId(x, y) { return x < y ? x + '_' + y : y + '_' + x; }
function areFriends(ownerId) {
  return get(/databases/$(db)/documents/friendships/$(friendshipId(ownerId, request.auth.uid)))
           .data.status == 'accepted';
}
function inCircle(ownerId, groupId) {
  return exists(/databases/$(db)/documents/users/$(ownerId)/groups/$(groupId)/members/$(request.auth.uid));
}

match /users/{ownerId}/profile/{bucketId} {
  allow read: if isOwner(ownerId);                                   // владелец видит всё
  allow read: if verified() && bucketId == 'everyone';               // дёшево: без обращений к базе
  allow read: if verified() && bucketId == 'friends' && areFriends(ownerId);
  allow read: if verified() && bucketId != 'private' && inCircle(ownerId, bucketId);
  allow write: if isOwner(ownerId);
}
```

Каждый `get()`/`exists()` — дополнительное тарифицируемое чтение; в одном запросе их не больше десяти.
Здесь используется одно.

### 3.5. Смена видимости свойства — атомарная операция

Изменить видимость = переписать `visibility` в корне и **перераспределить свойство по бакетам**:
удалить из тех, где оно больше не должно быть, добавить туда, где должно. Всё — одним `WriteBatch`
(Firestore гарантирует его атомарность). Иначе свойство может остаться видимым тем, у кого доступ
отобрали.

Раскладка и склейка — чистая функция, не зависящая от Firestore. Она вынесена в
`src/lib/model/visibility.ts` и покрыта тестами: именно здесь легче всего допустить утечку.

> ⚠️ **Чего правила проверить НЕ могут:** что содержимое бакетов соответствует карте `visibility` в
> корне. Пользователь технически может положить своё свойство в `profile/everyone`, объявив в
> `visibility` «никому». Но это раскрытие **собственных** данных **самим владельцем** — вреда другим
> нет. Принимаем осознанно.

### 3.6. Оценки по осям (точка человека в пространстве)

```
points/{uid}                    # служебное: dirty, updated, lastSync. Читает владелец и вычислитель.
points/{uid}/dims/{dimId}       # { value: 0…10 }   ← документ на ось
```

Ось = отдельный документ. Это снимает костыль 1.x «одна ось за обновление» (Ф3): правило валидирует
каждый документ по отдельности, `value is int && value >= 0 && value <= 10`. Заполнение профиля из
N осей стоит N записей — ровно столько же, сколько и раньше, но без искусственного ограничения на
последовательность.

**Оценки не видит никто, кроме владельца и вычислителя** (Т6). Настройка видимости на них не
распространяется: другой человек видит только *результат* — похожесть, — но не то, из чего она
сложилась. Вычислитель читает оси через Admin SDK (правила не применяются) запросом по группе коллекций.

### 3.7. Связи

```
relations/{uid}                 # читает ТОЛЬКО владелец; пишет ТОЛЬКО вычислитель
  computedAt, version
  top: [ { guestUid, similarity, proximity, commonality, … } ]   # настоящие имена, не a…t
```

`guestEmail` из документа **исчезает** — это чужие ПДн в чужом документе (`researches/03`).
Для карточки достаточно `guestUid` → `users/{guestUid}/profile/everyone`.

### 3.8. Очередь пересчёта вместо флага-строки

В 1.x `sync_status` — строка, и вычислителю приходилось сканировать всю коллекцию. Заменяем на
`points/{uid}.dirty: bool` + индекс: вычислитель делает `where('dirty', '==', true)` и берёт только
тех, кого надо пересчитать.

### 3.9. Права администратора

Список email в правилах (25 повторений в 1.x) заменяется на **custom claim** Firebase Auth:
`request.auth.token.admin == true`. Коллекция `admins` перестаёт быть мёртвой — из неё выдаются claim'ы.

## 4. Как это решает проблемы модели 1.x

| Проблема из `researches/02` | Что делаем |
|---|---|
| 1. Админ зашит списком email | custom claim `admin` (§3.9) |
| 2. Обфусцированные имена `a…t` | настоящие имена в `relations` (§3.7); декодер для миграции — `src/lib/similarity/legacy.ts` |
| 3. Правила = схема + валидация + авторизация | схема живёт в TypeScript; правила проверяют только права и границы |
| 4. Профиль читает любой аутентифицированный | **решено требованием Т1**: бакеты по кругам (§3.1) |
| 5. `email` дублируется в трёх коллекциях | ключ только `uid`; email нигде не хранится |
| 6. «Одна ось за обновление» | ось = документ (§3.6) |
| 7. Мёртвая подколлекция `privat/` | удаляется; её роль исполняет `profile/private` |
| 8. Нет истории оценок | **осознанно не делаем** (Т3) |
| 9. `sync_status` как флаг | очередь через `dirty` + индекс (§3.8) |

## 5. Уважительная миграция (набросок, детали — в `plans/`)

Соответствие старого новому:

| 1.x | 2.0 |
|-----|-----|
| `users/{uid}` (плоский профиль) | `users/{uid}` (корень) + `profile/everyone` |
| `users/{uid}.email` | не переносится (есть в Firebase Auth) |
| `ndimids/{id}.user_dims` (map) | `points/{uid}/dims/{dimId}` (документы) |
| `ndimids/{id}.sync_status` | `points/{uid}.dirty` |
| `relations/{id}` с полями `a…t` | `relations/{uid}.top[]` с именами (через `decodeLegacyRelation`) |
| `dims/{id}` | без изменений |
| `suggestions/{id}` | без изменений |
| `users/{id}/privat/` | удаляется (пусто) |
| `admins/{id}` | источник custom claims |
| — | новые: `friendships/`, `users/{uid}/groups/`, `users/{uid}/audience/` (при миграции пусты) |

**Видимость по умолчанию при миграции (Т7).** Владелец решил: всё остаётся видимым всем, как в 1.x.
Значит каждое свойство получает `visibility: 'everyone'` и целиком укладывается в `profile/everyone`.
Круги и дружба при миграции пусты — люди заведут их сами. Статус-кво сохраняется: ни одно поле не
раскрывается сверх того, что уже было видно, и ни одно не прячется молча.

**Принципы:**
1. Сначала на **копии** боевой базы (экспорт → отдельный проект), сверка числом документов и
   контрольными суммами.
2. Скрипт миграции **идемпотентен** и **обратим**: старые коллекции не удаляются до подтверждения.
3. Ни один человек не должен ничего заполнять заново.
4. Пересчёт связей после миграции обязан дать **те же числа**: паритет ядра проверен тестами
   (`src/lib/similarity/`).

## 7. Открытые вопросы реализации (решает агент)

- **Лимит кругов — 10** (плюс дефолтные «Друзья»). Проверяется правилом через `groupCount` в корне
  или счётчиком; выбрать при написании правил.
- **Дублирование свойства по бакетам** ограничено 11 документами. Обновление значения — `WriteBatch`.
  Убедиться, что 500 операций батча хватает: 5 свойств × 11 бакетов = 55. Хватает.
- **Согласованность бакетов** не проверяется правилами (§3.5). Чистая функция раскладки/склейки
  (`src/lib/model/visibility.ts`) покрыта тестами, включая проверку на расхождение значений.
- **Удаление круга** должно удалить одноимённый бакет профиля и почистить `audience/*`.
  Это многодокументная операция → кандидат на задачу вычислителя, если батч не влезает.
