## Коллекции — обзор

| Коллекция | Кто пишет | Кто читает | Роль |
|-----------|-----------|-----------|------|
| `admins/{adminId}` | админ | админ | список администраторов |
| `users/{userId}` | владелец профиля | любой аутентифицированный | публичный профиль человека |
| `users/{userId}/privat/{id}` | владелец | владелец | приватные данные. **Не используется** (заглушка) |
| `dims/{dimId}` | **только админ** | подтверждённая почта | измерение (ось) + его рейтинг |
| `ndimids/{ndimId}` | владелец профиля | владелец | **точка человека в пространстве**: оценки по осям |
| `relations/{relation}` | владелец + админ | **только владелец** | посчитанные связи: топ-250 похожих |
| `space/metrics` | админ | подтверждённая почта | агрегаты по платформе |
| `space/public_metrics` | админ | **все, без авторизации** | публичная витрина |
| `suggestions/{id}` | любой подтверждённый | **только админ** | заявка на новую ось |

## `users/{userId}` — профиль

```
uid          string   = auth.uid
email        string   = auth.token.email
gender       'm' | 'w' | 'nb' | 'null' | null
avatar       bool | null            # есть ли аватар в Storage
agreements   { agreement: true }    # согласие обязательно == true
born         { year: 1525..10000|null, month: 1..12|null, day: 1..31|null }
name         { first, middle, last, nick }   # каждое — { ru: str≤50, en: str≤50 }
                                             # nick — до 100 символов
about        { ru: str≤5000, en: str≤5000 }
settings     { language: 'ru'|'en', is_ios_pwa_shown: bool, is_mac_pwa_shown: bool }
time         { created: ts, last_sign_in: ts, ndim_id_updated: ts|null }
```

Читать может **любой аутентифицированный пользователь** — профиль публичен внутри платформы.

## `ndimids/{ndimId}` — точка человека в пространстве измерений ★

Самая важная коллекция: именно между этими документами считаются связи.

```
owner_uid    string   = auth.uid
owner_email  string   = auth.token.email
user_dims    map<dim_id, number 0..10>   # координаты человека по выбранным осям
sync_status  'never' | 'unsynchronized' | 'synchronized'
meta         { affected_dim_id: string(<30) | null }   # какая ось изменилась последней
time         { created: ts, updated: ts, last_sign_in: ts, last_sync: ts|null }
```

**Инвариант правил:** за одно обновление можно изменить **ровно одну ось** (или ноль):
`user_dims.diff(resource.data.user_dims).affectedKeys().size() <= 1`, и изменённая ось обязана
совпадать с `meta.affected_dim_id`, а значение — лежать в `0..10`.

Читать может только владелец (или админ). `sync_status` помечает, нужно ли пересчитать связи.

## `dims/{dimId}` — измерение (ось)

Пишет **только админ** — правила не описывают поля. Из `adminFunctions.calculateRatings()`:

```
stars    number   # сумма всех оценок, поставленных себе людьми по этой оси
rates    number   # сколько людей заполнили эту ось
rating   number   # stars / rates, округлённое до 0.1 — «средняя оценка оси»
...      # плюс тексты названия/описания оси (ru/en) — в правилах не описаны
```

`rating` показывается в интерфейсе рядом с осью (`app.js`, карточка измерения).

## `relations/{relation}` — посчитанные связи

```
owner_uid    string
owner_email  string
sync_status  'never' | ...
last_sync    ts | null
v            number | null      # версия расчёта
relations    <результат>        # топ-250 наиболее похожих на владельца
```

**Приватность:** читает только `owner_uid`. Похожесть — не публичная величина.
При `create` пользователь обязан создать документ **пустым** (`v == null`, `relations == null`);
заполняет его админ-вычислитель. При `update` пользователь **не может** менять `relations`, `v`,
`sync_status`, `last_sync` — все они обязаны совпадать с текущими значениями.

Внутри `relations` — метрики из `calculateRelation`, сохранённые под обфусцированными именами `a…t`
(см. `PROJECT_ARCHITECTURE_INTERNAL_MAP.md` → «Математическое ядро»). Это долг: имена надо вернуть.

## `space/metrics` и `space/public_metrics`

Агрегаты платформы: диаметр общего пространства (`calculateSpaceDiameter` = `√(число осей) · 10`),
счётчики пользователей и измерений. `public_metrics` читается **без авторизации** — это витрина на
лендинге.

## `suggestions/{id}` — заявка на новую ось

```
created      ts
author       string  = auth.token.email
description  string  5..300 символов
```

Так пространство измерений растёт снизу: оси предлагают люди, админ их модерирует и заводит в `dims`.

## Что в этой модели плохо (кандидаты на переделку в 2.0)

1. **Права админа зашиты списком email прямо в правилах** — `["kotkrinik@yandex.ru",
   "kotkrinik@gmail.com"]` повторяется ~25 раз. Коллекция `admins` при этом существует, но правилами
   не используется. Правильно: custom claims Firebase Auth или чтение `admins`.
2. **Обфусцированные имена метрик `a…t`** в `relations`. Данные нечитаемы без «матрицы соответствия»
   в коде.
3. **Правила — это и схема, и валидация, и авторизация одновременно.** Огромные выражения `allow
   update` (по 40+ условий) невозможно поддерживать. Схема должна жить в типах (TypeScript), а правила
   — проверять только права доступа и границы.
4. **`users` читает любой аутентифицированный.** Профиль с датой рождения и текстом «о себе» доступен
   всем зарегистрированным. Для платформы знакомств это спорно.
5. **`email` дублируется** в `users`, `ndimids`, `relations` (`owner_email`) — при смене почты
   разъедется. Ключом должен быть только `uid`.
6. **Ограничение «одна ось за обновление»** делает заполнение профиля из N осей ровно N записями в
   Firestore. Дорого и медленно.
7. **`users/{id}/privat/`** — мёртвая подколлекция-заглушка.
8. **Нет истории оценок.** Изменил оценку — прежнее значение потеряно. Для системы, которая ищет
   людей по близости, динамика человека во времени может быть ценной.
9. **`sync_status` как строка-флаг** вместо очереди задач: вычислитель обязан сканировать всю
   коллекцию, чтобы найти, кого пересчитывать.

## Решения владельца по новой модели (чат, 2026-07-10)

| Тема | Решение |
|------|---------|
| **История оценок** | ❌ **Не делаем.** «История — сложно, пока не делаем.» Проблема №8 ниже снимается: оценка перезаписывается, прошлое не хранится. |
| **Видимость личных данных** | ✅ **Настройка по каждому свойству в отдельности**, четыре аудитории: **всем · никому · друзьям · группе**. Проблема №4 решается этим. |

Второе решение вводит в продукт две новые сущности, которых в 1.x не было: **друзья** и **группы**.
Проработка — `researches/04_data_model_2x_proposal.md`; развилки уровня владельца —
`interviews/interview_002_visibility_friends_groups.md`.

## Требование владельца: миграция должна быть уважительной

Люди уже заполнили свои измерения. Это их труд и их данные. Любая новая модель обязана:

- перенести **все** существующие профили, оси и оценки без потерь;
- не потребовать от человека заново заполнять то, что он уже заполнил;
- быть обратимой (иметь возможность откатиться), пока миграция не подтверждена;
- сначала выполняться на копии данных и проверяться, и лишь затем — на боевых.

Проектирование новой модели и плана миграции — отдельная задача фазы 2/3. Развилки уровня владельца
(что показывать публично, хранить ли историю оценок) закрываются через `/interview`.
