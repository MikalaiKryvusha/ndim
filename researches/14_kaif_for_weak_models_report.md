# Исследование 14 — Отчёт-рекомендации для KAIF: как выжать качество из слабых моделей

> **Кому.** Владельцу и ИИ-агенту проекта KAIF. **Откуда.** Живая сессия NDim Space 2026-07-24:
> идея владельца → разведка старого кода → переработка сервера синхронизации → тесты → судья →
> боевой выкат. Полный след: `ideas/14_DONE`, `researches/13`, коммиты `dea6f53…77ae123`.
> **Вопрос владельца.** Какими средствами KAIF заставить слабые модели (класс Opus) работать
> качественнее — пусть медленнее, но с минимумом переделок для вайбкодера-владельца.
>
> Общий принцип всех рекомендаций: слабая модель хуже всего там, где надо ДОДУМЫВАТЬ
> (архитектуру, чужие намерения, состояние прода). Значит, фреймворк должен системно
> **заменять додумывание наблюдением**: документ вместо памяти, факт вместо догадки, прогон
> вместо «должно работать». Это всегда цена в скорости — и всегда выигрыш в переделках.

---

## 1. «Разведка до кода» — из привилегии багов в канон любой задачи с внешней правдой

**Факт сессии.** Владелец сказал: «в 1.x я экономил запросы, а 2.0 словно не вдохновлялся старым
кодом». Вместо кодинга по памяти агент снял устройство синхронизатора 1.x с живого исходника в
`researches/13` — и реализация после этого стала почти механической: архитектура (кэш, дельта,
ленивый контракт) уже была изобретена автором, оставалось перенести.

**Рекомендация KAIF.** Сейчас разведка формализована только для багов (`/bug-research`). Сделать
общий канон: **если у задачи есть внешняя правда** (старая система, чужой API, поведение прода,
док вендора) — **первый артефакт задачи всегда разведдок в `researches/`**, код запрещён до его
появления. Для слабой модели разведдок — это якорь, который вытесняет галлюцинацию: она кодит
«по документу», а не «по представлению». Бонус: разведдоки переиспользуются всеми будущими
сессиями (наш `researches/13` уже сэкономит часы любому следующему агенту).

**Цена/выигрыш.** +30–60 минут на задачу; минус целый класс переделок «сделал не то, что было».

## 2. Судья — перед каждым push и деплоем, с охотой на «диффы, которых агент не писал»

**Факт сессии.** Судейский проход поймал настоящую беду: `npm install --package-lock-only`
втихую добавил в манифест вычислителя зависимость `ndim-space: file:..` — боевая Docker-сборка
упала бы в crash-loop. Агент этого диффа «не писал» — его сгенерировал инструмент, и глазами
он не проверялся, пока судья не сверил дифф с заявленным объёмом работ.

**Рекомендация KAIF.** (а) Судья (`/fable-judge`) обязателен не только перед «done» беклога, но
**перед каждым push и каждым деплоем** — это дешёвая точка, где ещё всё откатывается. (б) В
чек-лист судьи добавить отдельную категорию: **«файлы, изменённые инструментами, а не рукой
агента»** (lock-файлы, манифесты, сгенерированное, авто-форматирование) — читать их дифф
построчно; слабые модели доверяют своим инструментам ещё слепее, чем себе. (в) Правило для
исполнителя: `git diff --stat` перед каждым коммитом, и всё, чего не намеревался менять, — стоп.

**Цена/выигрыш.** +5–10 минут на цикл; ловит именно тот сорт дефектов, который «зелёные тесты»
не видят (наш и был невидим для всех 29 тестов).

## 3. Запрет молчаливого изменения тестов + правило «тест должен доказывать то, что заявляет»

**Факт сессии.** При добавлении «окна новичка» старые тесты тихого периода стали бы проходить
**по неправильной причине** (вера — новичок, её больше не откладывают). Агент перестроил
фикстуры (тихий период теперь стережётся на «бывалом» Олеге, полный проход — на «бывалом»
Грише) и объяснил это в сообщении коммита. Слабая модель в этой точке обычно делает обратное:
подгоняет тест под новое поведение и рапортует зелёное.

**Рекомендация KAIF.** Конвенция + проверка судьёй: **любой дифф в `*.test.*` требует в
коммите блока «почему тест изменён и что он теперь стережёт»**. Изменение теста без
обоснования = фрод по умолчанию (у fable-judge это уже категория weakened checks — усилить её
формальным требованием к коммиту, чтобы проверка стала механической). Дополнительное правило
ревью: после изменения поведения спросить «не проходят ли старые тесты по новой причине?» —
это ровно та проверка, которую слабая модель сама не делает никогда.

**Цена/выигрыш.** Минуты; закрывает самый документированный провал агентов — тихую подгонку
тестов.

## 4. Правило одного шага: маленькое изменение → полный гейт → коммит

**Факт сессии.** Сессия дала шесть маленьких коммитов, и каждый проходил полный гейт
(test:calc + юниты + typecheck + kaif:check). Когда судья нашёл беду — откат стоил один файл,
а не день работы. Инцидент чинился коммитом `226aa94` поверх `ba7a9fd`, ничего не потерялось.

**Рекомендация KAIF.** В автономных циклах (`/autoloop`, `/dayloop`, `/nightloop`) закрепить:
**одно осмысленное изменение = один прогон всех гейтов = один коммит**. Слабой модели запретить
«пакетные» коммиты на полдня работы: большой дифф она уже не способна честно отревьюить сама, и
владелец потом переделывает всё скопом. Медленнее — да; зато каждая точка истории откатываема
и судима по отдельности.

**Цена/выигрыш.** Гейты гоняются в разы чаще; переделка в худшем случае — один шаг, не сессия.

## 5. «Зелёные тесты ≠ работает»: живой смоук + числовые инварианты до/после

**Факт сессии.** После 29 зелёных тестов агент всё равно: (а) прогнал службу живьём на
эмуляторе и читал её лог глазами; (б) проверил внутри свежего Docker-образа, что все модули на
месте (`IMPORTS_OK` — страховка от EXP-0040, когда образ отстал от кода и уронил бой);
(в) при выкате сверил доменные числа до/после: 93 человека, 5112 измерений, 2340 связей —
совпали, значит математика не тронута.

**Рекомендация KAIF.** Формализовать три дешёвых пост-гейта деплоя, посильных даже слабой
модели: **(1) смоук живого процесса с чтением лога первого цикла глазами; (2) проверка
самодостаточности артефакта деплоя** (образ/бандл стартует в изоляции); **(3) инварианты
домена** — перед работой выписать в STATUS числа, которые не должны измениться (счётчики,
суммы, размеры), после — сверить. Сверка чисел — операция, в которой слабая модель не
ошибается, а сигнал у неё высочайший.

**Цена/выигрыш.** +10 минут на выкат; ловит класс «тесты зелёные, прод мёртв».

## 6. Деплой-зеркало: снять фактическую конфигурацию прода до его замены

**Факт сессии.** Перед заменой боевого контейнера агент сделал `docker inspect` работающего —
и обнаружил, что прод живёт с интервалом 3600 с (вот откуда «полные синхронизации каждый час»,
на которые жаловался владелец!), а compose-дефолт вообще целится в эмулятор
(`host.docker.internal:8181`) — слепой перевыкат «по документации» отправил бы боевой сервер
в несуществующий эмулятор. Каждое отличие нового запуска от старого было осознанным решением.

**Рекомендация KAIF.** В ритуал деплоя (скилл `/release` или отдельный deploy-чеклист):
**(1) снять фактическую конфигурацию работающего прода** (inspect/env/версия) **до** замены;
**(2) явно перечислить каждое отличие нового запуска от старого с обоснованием; (3) после
выката обновить в репо документ «боевой запуск»** — единственный источник правды (наш
`plans/04` знал про 3600, но уже год как никем не сверялся). Прод-конфиг, живущий только в
запущенном процессе, — мина, на которой слабая модель подорвётся первой.

**Цена/выигрыш.** +5 минут на деплой; исключает «перевыкатил и незаметно сменил поведение».

## 7. Секция «Решения, принятые без владельца» — обязательная в якоре задачи

**Факт сессии.** Идея 14 закрывалась с явным списком: какие из открытых вопросов владельца
агент закрыл сам и как (свежесть, сердцебиение, интервал). Владелец отревьюил это за секунды —
и его следующий вопрос («а быстрые пересчёты новичкам?») попал ровно в одно из таких решений,
которое тут же стало отдельной фичей.

**Рекомендация KAIF.** В шаблоны идеи/бага/плана добавить обязательную секцию **«Решения,
принятые агентом без владельца»** (заполняется при закрытии). Слабые модели молча принимают
десятки микрорешений — эта секция принуждает их выложить допущения на стол, где владелец
дёшево ловит расхождения с видением ДО того, как на них наслоится следующая работа. Это же
лучший генератор правильных вопросов от владельца (проверено этой сессией).

**Цена/выигрыш.** Пара абзацев на задачу; переделки из-за «сам решил и никому не сказал»
превращаются в одну строку правки.

## 8. Документ-якорь: один на задачу, кодить — цитируя его

**Факт сессии.** Вся цепочка (жалоба → диагноз → разведка → реализация → решения → DONE)
живёт в одном документе `ideas/14` со ссылками наружу. Любая будущая сессия восстановит
контекст за одно чтение — включая ПОЧЕМУ сделано именно так.

**Рекомендация KAIF.** Канон уже есть — добавить принуждение для слабых моделей: в скиллах
исполнения требовать **перед кодом процитировать строку якоря, которая сейчас реализуется**
(«кодишь — цитируй план»). Модель, не способная назвать, какой пункт документа она делает,
делает отсебятину — и это обнаруживается ДО диффа, а не после.

**Цена/выигрыш.** Почти бесплатно; режет дрейф скоупа — главный источник «переделай всё».

## 9. EXPERIENCE — уроки в форме готовых приёмов, а не мемуаров

**Факт сессии.** Урок EXP-0050 записан с конкретикой: «считай запросы, а не сложность кода»,
«diff требует детерминированной сериализации», «иди в ndim_old и снимай механику дословно».
Прошлые уроки (EXP-0040 про отставший Dockerfile) прямо предотвратили повтор инцидента.

**Рекомендация KAIF.** Заточить формат урока под слабого читателя: обязательные поля **«как
воспроизвести проверку» (готовая команда)** и **«триггер-теги»**, по которым урок обязан
всплыть перед задачей. Слабая модель не сделает вывод из эссе, но исполнит копипастабельную
команду. Судья может проверять: перед задачей агент грепнул EXPERIENCE по тегам задачи и
процитировал релевантные уроки (или честно сказал «релевантных нет»).

**Цена/выигрыш.** Дисциплина записи; уроки начинают работать на модели, которые сами бы их
не вспомнили.

## 10. Недетерминизм — враг диффов: правило канонического порядка

**Факт сессии.** Первый боевой полный проход переписал 77 топов из 93 при неизменных данных —
из-за недетерминированного порядка людей с равной похожестью экономия diff-записи дырявилась.
Одна строка тай-брейка по uid закрыла дыру навсегда.

**Рекомендация KAIF.** В библиотеку уроков/чеклист ревью: **всё, что сравнивается или
кэшируется, обязано иметь канонический порядок** (сортировка с полным тай-брейком,
детерминированная сериализация с сортировкой ключей, запрет на `Date.now()`/random в
сравниваемом). Слабая модель сама не заметит недетерминизм никогда — он проявляется только
на живых данных; пункт чек-листа заметит вместо неё.

**Цена/выигрыш.** Строка кода там, где нужна; без неё — «зелёное, но зря переписывает всё»
незаметно живёт годами.

## Сводка: что усилить в KAIF в первую очередь

| # | Механика | Куда в KAIF | Эффект на слабой модели |
|---|----------|-------------|--------------------------|
| 1 | Разведдок до кода при внешней правде | новый канон + обобщение `/bug-research` | кодит по документу, не по галлюцинации |
| 2 | Судья перед push/деплоем + охота на диффы инструментов | `/fable-judge`, ритуалы циклов | ловит невидимое тестам |
| 3 | Обоснование любого изменения тестов | конвенция коммита + судья | закрывает подгонку тестов |
| 4 | Один шаг = полный гейт = коммит | `/autoloop`/`/dayloop`/`/nightloop` | откат шага, не сессии |
| 5 | Смоук + самодостаточность артефакта + инварианты чисел | деплой-чеклист | «зелёное ≠ работает» закрыто |
| 6 | Деплой-зеркало + прод-конфиг в репо | `/release`/деплой-чеклист | конец слепых перевыкатов |
| 7 | «Решения без владельца» — секция в якоре | шаблоны ideas/bugs/plans | допущения на столе владельца |
| 8 | «Кодишь — цитируй план» | скиллы исполнения | режет дрейф скоупа |
| 9 | Уроки с готовыми командами и триггер-тегами | формат EXPERIENCE | уроки исполняются, не вспоминаются |
| 10 | Канонический порядок для всего сравниваемого | чеклист ревью | недетерминизм ловится списком |

Общая формула: **слабой модели нельзя доверить суждение, но можно доверить процедуру.**
KAIF выигрывает там, где превращает суждение («вроде готово», «тест можно поправить»,
«деплой как обычно») в процедуру с наблюдаемым результатом (разведдок, судья, гейт на шаг,
инвариант, зеркало прода). Скорость падает на десятки процентов, переделки — на порядок;
для вайбкодера-владельца это чистая экономия головной боли.

# Часть 2 — Уроки из корпуса багов NDim (боли владельца с Opus, волны 08–09 и ранние)

> Корпус: 41 баг-документ в `bugs/`; из них **~30 найдены лично владельцем руками** за
> четыре волны ручного теста (9 + 14 + 4 + 3). Это и есть измеренная цена слабой модели:
> владелец-QA работал приёмочным контролем вместо продукта. Вердикт владельца дословно —
> `bugs/09` пп. 14/16: «многие вещи оригинала проигнорированы и утеряны ИИ-агентами; 2.0
> стал более сломанным и топорным, чем рукописный оригинал». Внутрипроектный ответ уже дан
> (`plans/06_quality_process_overhaul.md`); ниже — обобщение до механик УРОВНЯ ФРЕЙМВОРКА,
> переносимых в любой проект KAIF.

## Класс А. Секреты и ПДн в репозитории (bugs/01–03: списки email, keystore, SMTP-пароль)

**Три отдельных инцидента** — слабая модель тащит в git всё, что лежит рядом. Ответ NDim —
гейт `kaif:check` перед каждым push — уже стал частью KAIF. Дожать на уровне фреймворка:
**правило «сначала ignore, потом инструмент»** — любой инструмент/выгрузка/ключ появляется
в проекте ТОЛЬКО после строки в `.gitignore`; судья проверяет каждый новый бинарь/дамп в
диффе вопросом «почему это в git?». Секрет, пойманный гейтом, — успех процедуры; секрет,
пойманный владельцем, — провал фреймворка.

## Класс Б. Потеря канона: отсебятина вместо эталона (самый массовый класс)

**Факты:** выдуманные иконки вместо канонических (17), потерянная анимация загрузки (21),
пропавшие подсказки метрик (24), переставленные метрики (25), обеднённые карточки (27),
разросшееся меню (29), плоская история версий (30), исчезнувший вход по паролю (32),
пагинация пачками вместо прогрессивной подгрузки (13). Механизм ошибки (диагноз plans/06):
агент реализует по макету/плану и считает канон исчерпанным, хотя канон — **живое поведение
эталона**. Слабая модель заполняет пробелы плана изобретательством — молча.

**Механика KAIF:** там, где у проекта есть эталон (старая система, конкурент, бренд-бук), —
**инвентарь поведения как ИСЧИСЛИМЫЙ чек-лист паритета**, не проза: таблица «элемент →
поведение эталона → есть в новом? → OK/баг» (`researches/12` NDim — образец). Правило:
**нет строки инвентаря — нет кода**; судья мерит сдачу ПО СТРОКАМ инвентаря. Отличие от
разведдока Части 1: разведдок *описывает*, инвентарь *исчисляет* — слабой модели нужен
счётный список, описание она прочтёт и всё равно нафантазирует.

## Класс В. «Работает» без глаз: дефекты, видимые за минуту живого прогона

**Факты:** ноль анимаций на целых экранах (05 — криминалистика посчитала: 0 transitions,
0 keyframes на Профиле/Связях/Пространстве/Меню), прозрачные панели и меню (22, 23), синяя
обводка тапов (26), пляшущая лента (18), разновысокие карточки (15), пустота под панелью
(20). Ни один не пойман тестами — все пойманы глазами владельца за один проход.

**Механика KAIF (двойная):** (1) **DoD фреймворка**: живой прогон в браузере/на устройстве
со скриншотами (обе темы, две ширины) — артефакт сдачи, без него «готово» не произносится
(канон NDim с plans/06 — поднять в ядро KAIF). (2) Специфично для слабых моделей —
**считаемые прокси качества**: криминалистика бага 05 показала, что «топорность» измерима
(счётчик transitions/keyframes на экран = 0). Слабая модель не видит красоту, но отлично
считает: гейт «на новом экране ≥ N переходов, 0 — стоп», «прозрачность панелей
проверена скриптом», «размер бандла не вырос на X%». Прокси не заменяет глаз владельца —
он ловит нули до владельца.

## Класс Г. Выдуманные данные на лице продукта (bug 07: «С нами уже 2 184 человека» при 331 живом)

Слабая модель ставит правдоподобный литерал-заглушку и рапортует «готово». На лице
добро-проекта это ложь людям.

**Механика KAIF:** правило продукта: **любое число/имя/факт на UI обязан иметь источник**
(документ данных, канон, слово владельца) — плейсхолдер без источника = баг по определению;
«выдуманное число хуже отсутствующего» (формулировка NDim — в канон фреймворка). Судье —
охоту на «литералы, похожие на данные» в диффах UI.

## Класс Д. Точечный фикс вместо класса дефекта

**Факты:** владелец каждый раз формулирует «и в других контекстных меню» (23), «на всех
экранах» (21), «по всему приложению» (26) — потому что агент чинит найденный экземпляр, а
остальные вхождения оставляет владельцу. Из той же семьи — сегодняшний EXP-0040 (Dockerfile
копировал файл, а не директорию: следующий импорт снова уронил бы контейнер).

**Механика KAIF:** **правило полного обхода**: фикс начинается с инвентаря ВСЕХ вхождений
класса (греп/скрипт, список — в документ бага), чинится по списку, судья сверяет список, а
не экземпляр. «Починен один» ≠ «класс закрыт». Где возможно — фикс формой (директория
вместо файла, компонент вместо копий, токен вместо литерала), чтобы класс не мог разойтись
снова.

## Класс Е. Пачка багов от владельца — это один процессный инцидент

Сама форма боли — «я нашёл ПАЧКУ проблем» — важнее любого бага в пачке. NDim среагировал
правильно (вердикт 09.14/09.16 → `plans/06`: диагноз → лекарства-процессы → шаги), но это
было импровизацией сильной модели.

**Механика KAIF — институционализировать:** (1) волна ручного теста владельца с ≥N багами
автоматически заводит **процессный документ** по шаблону plans/06 («вердикт → честный
диагноз → лекарства → шаги с чекбоксами»), и агент обязан его исполнять параллельно фиксам;
(2) **метрика здоровья процессов фреймворка — размер следующей волны владельца**: если
волны не мельчают (9 → 14 → 4 → 3 в NDim — мельчают), лекарства не работают и пересматриваются.
Цель KAIF для слабых моделей формулируется именно так: **не «ноль багов», а «владелец
перестаёт находить их пачками»**.

## Сводка части 2

| Класс боли | Багов в корпусе | Механика KAIF |
|---|---|---|
| А. Секреты в git | 3 | гейт перед push + «сначала ignore, потом инструмент» |
| Б. Потеря канона | ~9 | исчислимый инвентарь паритета; нет строки — нет кода |
| В. «Работает» без глаз | ~7 | DoD с живым прогоном и скриншотами + считаемые прокси (0 анимаций = стоп) |
| Г. Выдуманные данные | 1 (яркий) | у каждого факта на UI — источник; плейсхолдер = баг |
| Д. Точечный фикс | ~4 | полный обход класса по списку; фикс формой |
| Е. Пачки как симптом | 4 волны | процессный документ на волну + метрика «волны мельчают» |

Части 1 и 2 сходятся в одном: слабая модель проваливается не в коде, а в **невыполненных
неявных обязательствах** (канон, полнота, честность данных, глаза). KAIF должен делать
неявное — явным и исчислимым: список, гейт, инвентарь, метрика. Всё остальное слабая
модель уже умеет.
