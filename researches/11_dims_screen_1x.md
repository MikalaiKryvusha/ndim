## 1. Главное открытие: «Все измерения» — это НЕ каталог

Плоский список из 5111 записей, отсортированный по алфавиту, — это не «упрощение» экрана 1.x.
Это **другой продукт**. В 1.x «Все измерения» — **лента неоценённого, в случайном порядке, с
бесконечной подгрузкой**.

```js
let all_dims_available_list = [];          // id измерений, которые человек ЕЩЁ НЕ ОЦЕНИЛ

function dimsRandomiser() {                // равновероятный выбор одного id
  return all_dims_available_list[Math.floor(Math.random() * all_dims_available_list.length)];
}

async function getOneRandomDimFromFirestoreToAllDimsPage() {
  const random_dim = dimsRandomiser();     // 1. взять случайный id
  // ... взять документ из кеша (dims_cache_map), иначе — из Firestore
  all_dims_available_list = all_dims_available_list.filter((id) => id !== random_dim);  // 2. ИЗЪЯТЬ
  parent.appendChild(createDimCardElement("all", snapshot, null));                      // 3. дорисовать
}

async function allDimsLoader() {           // подгрузка по одной карточке
  while (all_dims_available_list.length !== 0 && is_all_dims_loader_is_visible) {
    await getOneRandomDimFromFirestoreToAllDimsPage();
  }
}
```

`is_all_dims_loader_is_visible` выставляет **`IntersectionObserver`**: как только элемент-загрузчик
(«Загрузка ( )» внизу списка) попадает в окно, цикл продолжает добирать карточки. Это классический
бесконечный скролл, но **по одной штуке и случайно**.

### Почему это важно (и почему алфавит — это провал)

- **Непредсказуемость.** Человек, открывая экран, каждый раз видит новое. Это единственный способ
  показать 5111 измерений: любой фиксированный порядок означает, что хвост каталога не увидит
  никто и никогда.
- **Лента сама себя опустошает.** Оценил измерение → оно **уходит** из `all_dims_available_list`.
  Удалил оценку → возвращается (`all_dims_available_list.push(dim_id)` в удалении). Экран —
  это **работа, которая заканчивается**, а не бесконечная простыня.
- **Справедливость к измерениям.** Равновероятный выбор даёт шанс каждому измерению, а не только
  тем, чьё название начинается с «А» или у кого больше голосов.

> При алфавитной сортировке человек навсегда упирается в «…А в душе я танцую» и физически не
> способен добраться до остального. Владелец назвал это «бред!!!» — и это точный диагноз.

## 2. Две суб-вкладки и два разных порядка

| Вкладка | Что показывает | Порядок |
|---------|----------------|---------|
| **«Все»** | измерения, которые человек **ещё не оценил** | **случайный**, по одной, бесконечная подгрузка |
| **«Мой NDim ID»** | измерения, которые человек **оценил** | **по убыванию СВОЕЙ оценки** (от 10 к 0) |

Комментарий из кода (`myDimsLoader`): *«получать по порядку от самых высокооценённых dims до
менее оценённых»*. Пустой NDim ID → подсказка `my_ndimid_is_empty`.
Кончились доступные измерения → подсказка `there_is_no_more_dims_in_ndim_system`.

## 3. Анатомия карточки (шаблон `dims_card_template`)

Карточка — **`<details>/<summary>`**, то есть раскрытие штатное, а не самодельное.
**Каждое измерение — отдельная карточка** с зазором, а не строка в общем списке.

### Свёрнуто (`<summary>` — видно всегда)

1. **Бейдж «Новое 🔥»** — если `time.created` моложе **14 дней** (`two_weeks_ago`).
2. **Тип** — `dim.type[lang]`, с заглавной буквы и двоеточием: **`Роман:`**, `Фильм:`, `Видеоигра:`.
3. **Название** — `dim.name[lang]`, в кавычках, и **год**: `"Лолита" (1955)`.
4. **Меню `⋮`** — три действия: «Искать в Интернете» · «Сообщить об ошибке» ·
   «Удалить из NDim ID».
5. **Рейтинг сообщества** — блок `dim_rating_container`, **скрыт целиком, если голосов нет**:
   - крупное **значение** (`9.5`) — на кадре зелёное;
   - графика звёзд;
   - **число голосов**: `(4 голоса)` — с русской морфологией.
6. **Одиннадцать звёзд 0…10** (`dim_card_bottom` → `dim_card_stars_container`) — **видны в
   свёрнутом состоянии**. Это и есть жест оценки: `<div class="dim_card_star" value="N">` с
   картинкой звезды и подписью-цифрой под ней.

### Раскрыто (клик по названию)

Добавляются: **Описание** · **Автор** · **Год** · **Теги**. Звёзды остаются на месте.

## 4. Что из этого у нас УЖЕ есть, а чего нет

| Что | В боевой базе 2.0 | В коде 2.0 |
|-----|-------------------|-----------|
| `title` (name), `description` | ✅ есть | ✅ читается |
| `stars`, `rates`, `rating` | ✅ есть (посчитаны миграцией `05-dims-catalog`) | ✅ читается |
| **`type`**, **`author`**, **`year`**, **`tags`** | ✅ **целы** (миграция писала `title` ДОБАВЛЕНИЕМ, поля 1.x не тронуты) | ❌ **не объявлены в `DimDoc` и не читаются** |
| `time.created` (для бейджа «Новое») | ✅ есть | ❌ не читается |
| Лента случайного неоценённого | — (это поведение, не данные) | ❌ **нет: плоский алфавитный список** |

**Вывод: данные для честной карточки уже лежат в боевой базе.** Потерян не материал — потеряна
форма. Нужно расширить `DimDoc` и переписать экран, миграция данных не требуется.

## 5. Что переносим, а что переосмысливаем

**Переносим как есть (дань уважения):**
- отдельная карточка на измерение; тип + название + год в заголовке;
- рейтинг сообщества со звёздами и числом голосов — **в свёрнутом виде**;
- одиннадцать звёзд 0…10 с цифрами — **в свёрнутом виде**;
- раскрытие → описание, автор, год, теги;
- бейдж «Новое 🔥» (14 дней);
- **лента: случайный порядок + бесконечная подгрузка + изъятие оценённого**;
- «Мой NDim ID» — по убыванию своей оценки.

**Переосмысливаем:**
- **«Удалить из NDim ID»** в 1.x жило в меню `⋮`. В 2.0 у нас снятие оценки уже есть — решить,
  где ему жить (меню или крестик у своей оценки).
- **«Искать в Интернете»** — внешняя ссылка; полезна, но это отдельное решение владельца.
- **Загрузка по ОДНОЙ карточке** — в 1.x это следствие «один документ = один запрос». В 2.0 можно
  брать пачкой (например, по 20) и всё равно **сохранять случайность**: перемешать список
  доступных id один раз за сессию и выдавать порциями. Поведение для человека то же, запросов
  меньше. ⚠️ Случайность обязана быть **на списке id**, а не «случайная выборка из Firestore» —
  Firestore такого не умеет, и 1.x именно поэтому держал список id на клиенте.
