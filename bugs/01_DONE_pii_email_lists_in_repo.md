# Bug 01 — Персональные данные (299 email реальных пользователей) лежат в репозитории

**Статус:** ✅ DONE (2026-07-10) — изъяты из репозитория, публичная экспозиция закрыта
**Версия/сборка:** наследие NDim 1.x · **Когда/контекст:** 2026-07-10, при развёртывании KAIF и переносе `legacy/`

## ✅ СТАТУС: DONE (2026-07-10)

### ⚠️ Уточнение первоначальной оценки

Изначально агент решил, что адреса **сейчас** публично доступны. Проверка клона `ndim-old` показала,
что это **не так**: владелец ещё раньше переписал историю удалённого репозитория — коммит
`b107581 "Purge nodemailer/ from history; ignore it"`. В достижимой истории `ndim-old`:

```bash
git log --all --oneline -- nodemailer     # → 0 коммитов
# поиск файлов email_list* по всем деревьям всех коммитов → пусто
```

Списки почт жили только в **локальной** копии `D:\work\ndim`, которая отставала от GitHub на 3 коммита
и осталась на старой ветке истории. Публичная экспозиция была раньше и уже устранена владельцем;
её масштаб меньше, чем предполагалось при заведении бага.

### Что сделано

1. Списки почт вынесены из проекта в локальный `.private/` (в `.gitignore`, никогда не коммитится).
2. Новая история NDim Space 2.0 начата с чистого листа — файлов нет ни в одном коммите.
3. Владелец сделал старый репозиторий **приватным** и переименовал его в `ndim-old`.
4. `npm run kaif:check` теперь падает, если файл со списком почт окажется под контролем версий.

### Как проверено

```bash
gh repo view MikalaiKryvusha/ndim-old --json isPrivate,forkCount
# → { "isPrivate": true, "forkCount": 0 }
```

`forkCount: 0` — критично: форков нет, значит копий с персональными данными ни у кого не осталось.

**Остаточный риск:** адреса присутствуют в `.private/ndim-1.x-history.bundle` (старая ветка истории,
локально) и, теоретически, в «висячих» объектах GitHub после переписывания истории. После приватизации
репозитория они недостижимы извне.

**Остаточный вопрос (не баг):** уведомлять ли 299 участников о том, что их адреса были публичны.
Решение владельца, вынесено в `STATUS.md` → «Ожидает решения человека».

## Симптом

В репозитории открытым текстом хранятся списки электронных адресов реальных людей — участников
открытого бета-тестирования NDim Space:

| Файл | Размер | Адресов |
|------|--------|---------|
| `legacy/nodemailer/email_list_prod.txt` | 18 036 байт | **299** |
| `legacy/nodemailer/email_list.txt` | 201 байт | 1 |
| `legacy/nodemailer/email_blacklist.txt` | 37 байт | 2 |

Ожидаемо: персональные данные пользователей не находятся в системе контроля версий и тем более не
попадают в публичный репозиторий.

## Воспроизведение (детерминированное)

```bash
# посчитать адреса, не выводя их в терминал:
grep -c '@' legacy/nodemailer/email_list_prod.txt   # → 299
```

## Криминалистика

- Файлы **отслеживались git'ом** в оригинальном репозитории: `legacy/GIT_HISTORY.txt` и
  `legacy/ndim-1.x-history.bundle` содержат их во всех коммитах, где они менялись. Удаление файла из
  рабочего дерева историю не очистит.
- Согласно `GOAL.md`, оригинальный репозиторий публиковался на
  `https://github.com/MikalaiKryvusha/ndim`. Если он был или является публичным — **адреса уже
  раскрыты**, и их следует считать скомпрометированными.
- `legacy/nodemailer/mailer.js` читает эти списки и рассылает по ним письма через nodemailer.

## Корневая причина / Гипотезы

Рассылка приглашений в бету была сделана как разовый скрипт, а список получателей — как обычный
текстовый файл рядом с ним. Отдельного хранилища для персональных данных в проекте нет: единственная
база — Firestore, а список для рассылки собирался вручную. Классическая оплошность, а не злой умысел
(бритва Хэнлона).

## План фикса

Фикс **не в `legacy/`** — наследие только для чтения (`AGENT_GUIDE.md`). Действия относятся к новой
истории и к тому, что произойдёт до первого push.

1. **До первого push нового репозитория** — решить судьбу `legacy/nodemailer/*.txt`:
   - вариант «вычистить»: удалить списки из рабочего дерева и добавить в `.gitignore`, сохранив локальную
     копию вне репозитория;
   - вариант «оставить, но не публиковать»: держать репозиторий приватным.
   Это решение владельца, а не агента — затрагивает и приватность людей, и открытость проекта.
2. **Проверить, публичен ли `MikalaiKryvusha/ndim`.** Если да — адреса уже раскрыты; нужно решить,
   уведомлять ли пользователей, и убрать ли репозиторий в приватные.
3. **В NDim Space 2.0** рассылка не должна брать получателей из файла в репозитории. Адреса живут
   только в Firestore (коллекция `users`), доступ — через сервисный аккаунт, список получателей
   формируется запросом на лету.
4. `legacy/ndim-1.x-history.bundle` (37.8 МБ) содержит адреса в истории. Если репозиторий станет
   публичным, bundle тоже раскрывает их. Учесть при решении по п.1.
