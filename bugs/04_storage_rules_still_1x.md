# Bug 04 — Правила Firebase Storage всё ещё от версии 1.x

**Статус:** 🔧 открыт · **Заведён:** 2026-07-12 (агентом, по следам боевого выката)
**Важность:** средняя — это **долг**, а не дыра. Данные не утекают, но права живут не там, где должны.

## Симптом

Правила Storage в боевом проекте **ни разу не выкатывались из 2.0**. В `firebase.json` секции
`storage` нет вовсе — значит на проде до сих пор действует ruleset версии 1.x. Обнаружено, когда
искали, почему не показываются фотографии людей (`EXP-0043`).

Живые правила (прочитаны через Firebase Rules API):

```
match /admin/{file} {
  allow read, write: if request.auth != null && request.auth.token.email in [
    "kotkrinik@yandex.ru",
    "kotkrinik@gmail.com"
  ];
}

match /public/{file} {
  allow read: if true;
  allow write: if request.auth != null && request.auth.token.email in [ ...те же две почты... ];
}

match /users/{userId}/avatar/{file} {
  allow write: if request.auth != null && request.auth.uid == userId;
  allow read:  if request.auth != null;
}
```

## Что здесь не так

1. **Права админа зашиты списком email прямо в тексте правил.** Это ровно тот антипаттерн, который
   мы уже вычистили из Firestore: в 2.0 админ — это **claim в токене** (`isAdmin()`), потому что
   claim нельзя подделать записью документа, а список почт приходится править в двадцати местах.
   Здесь он остался жив. (В `researches/02` он числится среди дефектов модели 1.x — «права админа
   зашиты списком email прямо в `firestore.rules`, ~25 повторений».)
2. **Правила не знают про модель видимости 2.0.** Фотография в 2.0 — это свойство профиля, у него
   есть аудитория (`researches/04`: круги, бакеты). Сейчас же аватар читает **любой вошедший**,
   включая гостя, независимо от того, кому человек открыл своё фото. Пока это ПАРИТЕТ с 1.x (там
   аватары тоже видели все вошедшие), и миграция честно положила `avatar` в бакет `everyone`, —
   но как только человек сможет скрыть фото, правила начнут врать.
3. **Секция `storage` отсутствует в `firebase.json`** — то есть правила Storage не под контролем
   версий и не деплоятся вместе с остальным. Их нельзя проверить тестами, как `firestore.rules`
   (там **85** тестов отказов на эмуляторе).

## Чего НЕ надо делать сгоряча

Не «починить» правила, ужесточив чтение аватаров: сегодня на них держится единственный способ
показать людям лица (`src/lib/data/avatar.ts`). Сначала — правила и тесты, потом — ужесточение.

## Как чинить

1. Завести `storage.rules` в репозитории + секцию `storage` в `firebase.json`.
2. Перенести смысл: владелец пишет своё, читает — по аудитории свойства `avatar` (модель кругов),
   админ — по **claim**, а не по списку почт.
3. Покрыть тестами отказов на эмуляторе Storage — так же, как `tests/rules/rules.test.ts`
   покрывает Firestore. Проверить **мутациями**: снятое умолчание «запрещено» обязано ронять набор.
4. Выкатить (`firebase deploy --only storage`) и убедиться, что фотографии по-прежнему видны.
