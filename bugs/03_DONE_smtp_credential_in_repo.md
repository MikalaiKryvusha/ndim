# Bug 03 — 🔴 КРИТИЧНО: живой пароль SMTP в публичном репозитории

**Статус:** ✅ DONE (2026-07-10) — пароль отозван владельцем, литерал изъят из кода
**Версия/сборка:** наследие NDim 1.x · **Когда/контекст:** 2026-07-10, аудит секретов при развёртывании KAIF

## Симптом

`legacy/nodemailer/mailer.js` содержал захардкоженный пароль приложения Yandex для почтового ящика
`ndimspace@yandex.ru`, использовавшегося для рассылки приглашений в бету:

```js
// было (строки 15–23):
const transporter = nodemailer.createTransport({
  host: 'smtp.yandex.ru',
  port: 465,
  secure: true,
  auth: { user: 'ndimspace@yandex.ru', pass: '<16-символьный пароль приложения>' },
});
```

Файл отслеживался git'ом в репозитории `MikalaiKryvusha/ndim`, который, по подтверждению владельца
(`homeworks/01`, шаг 1), **публичен**. Значит, пароль был доступен любому.

## Оценка риска

🔴 **Высокий, в отличие от `bugs/01` и `bugs/02`.** Это не данные — это действующий доступ.
Обладатель пароля может:
- отправлять письма **от имени NDim Space** 299 участникам беты (фишинг под доверенным брендом);
- в зависимости от типа пароля — читать почтовый ящик проекта;
- скомпрометировать восстановление паролей и любую переписку, привязанную к этому адресу.

Пароль лежал в публичном доступе неизвестно сколько времени. **Считать его скомпрометированным.**

## Криминалистика

- Единственное место утечки: `legacy/nodemailer/mailer.js`. Проверено grep'ом по
  `pass\s*:|password\s*[:=]` во всём `legacy/`: остальные совпадения в `public/scripts/app.js` и
  `public/index.html` — это переменные и id полей формы (`let password = null;`,
  `manage_account_page_button_edit_password`), не секреты.
- Список получателей (`email_list_prod.txt`) содержит только `email`, язык, имя, число измерений —
  **без** `uid` и `secret_key`. Секретов пользовательских аккаунтов в утечке нет.
- Пароль присутствует также во всех коммитах истории 1.x → в `.private/ndim-1.x-history.bundle`.

## Что уже сделано (2026-07-10)

1. Литерал пароля изъят из `legacy/nodemailer/mailer.js`; код читает `process.env.SMTP_USER` и
   `process.env.SMTP_PASSWORD`. Логика рассылки сохранена как документация.
2. Списки почт и ключ подписи вынесены из репозитория в локальный `.private/` (в `.gitignore`).
3. `.private/` и `*.keystore`, `*.jks`, `*.pem`, `.env*` добавлены в корневой `.gitignore`.
4. Новая git-история не содержит ни пароля, ни ПДн, ни `bundle` (проверено перед первым коммитом).

> Это единственное исключение из инварианта «`legacy/` — только для чтения» (`AGENT_GUIDE.md`).
> Санация секретов прямо предписана владельцем (`homeworks/01`, шаг 4) и не является рефакторингом.

## ✅ СТАТУС: DONE (2026-07-10)

**Что сделано:**
1. Владелец **отозвал пароль приложения** в Yandex ID (подтверждено в чате 2026-07-10):
   *«ndimspace@yandex.ru я отозвал пароль приложения яндекс, он больше невалидный»*.
   Это устраняет риск полностью — даже если пароль у кого-то есть, он мёртв.
2. Литерал пароля изъят из `legacy/nodemailer/mailer.js`; код читает `process.env.SMTP_PASSWORD`.
3. Новая git-история не содержит пароля ни в одном коммите.
4. `.env*`, `*.keystore`, `*.jks`, `*.pem` и `.private/` внесены в корневой `.gitignore`.

**Как проверено:** `grep -i "pass\s*:|password\s*[:=]"` по всему `legacy/` — совпадения остались только
в `public/scripts/app.js` и `public/index.html`, и это переменные/id полей формы, не секреты.
Перед первым коммитом индекс проверен на отсутствие секретов.

**Остаточный риск: практически отсутствует.** Проверка клона `ndim-old` показала, что владелец ещё
раньше переписал историю удалённого репозитория (коммит `b107581 "Purge nodemailer/ from history"`), и
строки `smtp.yandex.ru` нет ни в одном достижимом коммите (проверено перебором всех деревьев всех
коммитов). Пароль жил в **локальной** копии `D:\work\ndim`, отстававшей от GitHub на 3 коммита, и в
`.private/ndim-1.x-history.bundle` (старая ветка истории, локально). После отзыва он бесполезен.
Экспозиция была короче, чем предполагалось при заведении бага.
