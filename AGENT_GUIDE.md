# NDim Space — Руководство ИИ-агента

Этот файл читается ИИ-агентом перед каждой задачей. Это **канон** проекта: правила, карта, команды,
конвенции. Держи его точным — свежая сессия агента с пустым контекстом полностью полагается на этот
документ, чтобы приступить к работе.

> 🧠 **ГЛАВНЫЙ ПРИНЦИП — ПРОСТОТА (читай `PHILOSOPHY.md`).** Если что-то занимает много времени, это
> НЕ трудная задача и НЕ баг библиотеки — агент ДЕЛАЕТ СЛИШКОМ СЛОЖНО, потому что НЕ ПОНЯЛ ЗАДАЧУ.
> Всё должно быть просто (KISS + Оккам). Застрял → перепойми задачу, найди встроенный простой путь,
> НЕ наращивай сложность. Застревание = «упрости своё понимание», а не «копай глубже».

> 🤖 **АВТОНОМНЫЙ РЕЖИМ.** Когда человек отошёл или дал автономию, активной интерактивной задачи нет,
> а в `STATUS.md` есть открытый автономный беклог — агент ДОЛЖЕН по собственной инициативе войти в
> подходящий цикловой скилл (`/autoloop`, `/dayloop` или `/nightloop`) и молоть беклог, коммитя
> прогресс и перезапуская себя после каждой задачи. Останавливайся только по стоп-условиям скилла.
> Не входи в цикл, если человек только что дал конкретную интерактивную задачу.

## Перед каждой задачей — чеклист

```
1. Прочитай STATUS.md            # текущее состояние: что сделано, где мы, что дальше
2. Вспомни опыт                  # грепни EXPERIENCE.md по тегам задачи — не повторяй известные тупики (скилл: /experience)
3. git status                    # что изменено, что не закоммичено
4. git log --oneline -5          # где мы в истории
5. Прочитай MEMORY.md (если есть) # профиль пользователя, ключевые решения
6. Грузи ТОЛЬКО нужный срез      # используй маршрутизатор контекста ниже — читай обязательный минимум + доки под тип задачи, а не всё подряд
7. Прочитай релевантный план     # plans/<feature>.md, если задача касается конкретной фичи
8. Проверь карту и радиус поражения # перед правкой кода: PROJECT_ARCHITECTURE_INTERNAL_MAP.md — кого заденет; обнови карту, если связи изменились
9. Запусти сборку (если трогаешь код)  # см. раздел «Сборка»
10. Используй испытательный стенд  # см. раздел «Стенд» — управляй софтом и наблюдай за ним без человека
11. Комментируй код              # блоки комментариев, классы, модули, важные строки
12. Рефлексируй по багам в bugs/ # один md на баг; следуй BUG_FIXING_FRAMEWORK.md
13. Фиксируй опыт                # после значимого успеха/провала допиши урок в EXPERIENCE.md (скилл: /experience)
14. Периодически перечитывай ключевые документы:
    - PHILOSOPHY.md   ← принцип простоты; если застрял, сначала сюда
    - AGENT_GUIDE.md
    - STATUS.md
    - BUG_FIXING_FRAMEWORK.md
    Правь их, когда это сделает будущую автономную работу эффективнее. Агент работает через
    сессии, теряющие контекст, — эти документы должны позволять свежей сессии стать продуктивной
    с нуля.
15. Рассказывай в чате, хотя бы немного, естественным языком — что ты делаешь прямо сейчас, — чтобы
    человек мог заглянуть и следить за ходом.
16. Документы от человека (идеи, баги, фичи): прочитай их, исправь опечатки, минимально
    переструктурируй в чистый структурированный вид для ИИ. После реализации по такому документу
    впиши в него статус и дату реализации.
```

→ **`STATUS.md`** — главный файл состояния. Обновляй его после каждой значимой задачи.

### Маршрутизатор контекста (прогрессивная загрузка) — читай только нужный срез

Не читай все документы «на всякий случай» — это забивает тот самый контекст, который ты пытаешься
беречь. Всегда читай **обязательный минимум**, затем только документы под тип задачи; остальное — по
требованию.

| Тип задачи | Читать (минимум сверх обязательного) |
|------------|--------------------------------------|
| **Обязательный минимум (всегда)** | `STATUS.md` · `PHILOSOPHY.md` (набор принципов) · этот маршрутизатор · `EXPERIENCE.md` (греп по тегу) |
| Баг | `BUG_FIXING_FRAMEWORK.md` · `bugs/<этот>` · карта (радиус поражения) |
| Фича / идея | `ideas/<эта>` · `MASTER_PLAN.md` · релевантный `plans/<этот>` |
| Рефакторинг / правка | `AGENT_GUIDE.md` · обе карты (радиус поражения) |
| Планирование | `MASTER_PLAN.md` · `GOAL.md` · открытый беклог |

Разделы этих документов заякорены — адресуй срез (`DOC.md#якорь`), а не перечитывай файл целиком.
Обязательный минимум **не подлежит** лени: `PHILOSOPHY.md` применяется всегда.

### Журнал опыта — `EXPERIENCE.md`

`EXPERIENCE.md` — растущий, удобный для грепа журнал уроков агента (внешняя память о том, что работает,
а что нет). **Вспоминай** релевантные записи перед задачей (греп по тегу); **фиксируй** короткий урок
после любого значимого успеха или провала — в циклах делай и то, и другое, не дожидаясь человека.
Скилл: `/experience`. Граница: `bugs/` = один документ на дефект; `EXPERIENCE.md` = короткие
межзадачные уроки уровня подхода (включая успехи). Живой справочник — никогда не помечается DONE.

## Цель проекта

NDim Space — веб-приложение для поиска похожих людей: честная система знакомств, которая находит
человека по математической близости в многомерном пространстве самооценок, а не удерживает
пользователя в иллюзии бесконечного выбора ради подписки. Сейчас проект переживает переход от
ручной поделки энтузиаста (версия 1.x, приватный архив `MikalaiKryvusha/ndim-old`) к NDim Space 2.0 —
современному веб-сервису, сделанному по best practice. Полное видение владельца — в `GOAL.md`.

## Архитектура — карта

Проект в точке перезапуска: продуктового кода 2.0 ещё нет. В корне живут только документы KAIF,
`tools/` и директории знаний. **Старого кода в проекте нет** — он удалён по решению владельца.

Целевая архитектура 2.0 (решения интервью №001):

```
        │                                   (SSR и домен = индексация поисковиками)
        ├──▶ Firebase Auth / Firestore
        │
ядро похожести (чистый TS-модуль, покрытый тестами)  ← переиспользуется и клиентом, и вычислителем

        └──▶ по расписанию читает Firestore, считает связи, пишет relations обратно
```

## Дизайн — правило четырёх макетов

Дизайн NDim Space 2.0 делается **новым**. От версии 1.x заимствуется только идея; интерфейс
придумывается заново — современный, удобный, лёгкий. Единственное, чем мы связаны, — существующие
данные пользователей (но и модель данных можно перепроектировать, см. ниже).

**Перед реализацией любого ключевого брендового или UI/UX-решения агент обязан:**

1. Нарисовать **4 варианта макета** экрана или компонента.
2. Показать их автору.
3. Автор утверждает один вариант и говорит, что в нём поправить.
4. Утверждённый вариант с правками принимается в работу.

Так — по **всем** ключевым UI/UX и брендовым решениям. Самое очевидное (отступ, порядок пунктов меню,
цвет ошибки по уже принятому гайдлайну) агент делает сам, без макетов.

### Форма экранов-сводок: дашборд из виджетов (решение владельца, 2026-07-12)

Владельцу нравится идея **приборной панели, наполняемой виджетами** — она принята как форма для
экранов-сводок (первый из них — «Пространство», `design/space-mockups.html`, V1).

- Экран собирается из **виджетов**: плитка-метрика, лента «Сегодня», сводка «Сейчас в
  Пространстве», «Версии», «Сервер синхронизации» и т. д. Каждый виджет — самостоятельный кусок,
  который можно переставить или убрать.
- В будущем — **наборы виджетов и настройка вида пользователем** (человек собирает себе панель сам).
- Но **дефолтное каноническое состояние — одно для всех**: оно проектируется и утверждается как
  обычный макет. Настройка вида не отменяет канона, она — надстройка над ним.

Практический вывод для агента: новые сводные экраны верстай **виджетной сеткой** (карточка = виджет),
а не монолитом. Тогда настройка вида не потребует переписывать экран заново.

**Чем рисовать (по возрастанию сложности):**
- Сразу в **SVG** или **HTML+CSS** — проще, красивее, сразу переиспользуется в коде. Растр получаем
  конвертацией, если он вообще нужен.
- Художественная, артистическая графика — не наша работа: ставим задачу ChatGPT напрямую либо заводим
  домашку автору (`homeworks/`), он нарисует.

## Модель данных

### ⚠️ КАНОН: экономить запросы к базе (принцип владельца, 2026-07-12)

> «Я весь первый NDim писал так, чтобы **ЭКОНОМИТЬ ЗАПРОСЫ К БАЗЕ!!!** Везде, где могу».

Это не придирка, а архитектурное решение автора, и 2.0 обязан его уважать: **Firestore берёт
деньги за КАЖДОЕ чтение документа.** Агент это уже нарушил — `loadProfileScreen` вычитывал **все
5111 документов каталога** при каждом открытии профиля.

Правила, вытекающие из принципа:

- **Никогда не читай коллекцию целиком, чтобы показать её кусок.** Каталог измерений читается
  ОДНИМ документом-индексом `dims/dims_list` (наследие 1.x: JSON-строка `{id: {ru, en, year}}`),
  а полные карточки берутся порциями по мере прокрутки. См. `src/lib/data/dims.ts`.
- **Кешируй прочитанное на сессию.** Каталог для человека неизменяем — перечитывать одно и то же
  незачем (в 1.x это `dims_cache_map`).
- **Считай, во что обойдётся экран.** Прежде чем писать `getDocs(collection(...))`, спроси: сколько
  это документов у 331 человека? Если ответ измеряется тысячами — решение неверное.
- Агрегаты (счётчики, статистика) производит **сервер синхронизации**, а не клиент: он и так
  обходит все точки, ему это дёшево.

### Проектирование

Владелец **разрешил спроектировать новую модель данных Firestore** — текущая, по его собственной
оценке, может быть неоптимальной. Но требуется **уважительная миграция** существующих данных на новую
модель: люди уже заполнили свои измерения, и этот труд нельзя потерять.

Точный слепок текущей модели — `researches/02_firestore_data_model_1x.md`. Там же перечислено, что в
ней плохо. Проектирование новой модели и плана миграции — задача фаз 2–3; развилки уровня владельца
(что показывать публично, хранить ли историю оценок) закрываются через `/interview`.

## Сборка

Стек: **SvelteKit + TypeScript** (Kit 2.69 · Svelte 5 · Vite 8). Инициализирован 2026-07-10.

```bash
npm run dev        # локальная разработка (vite dev, http://localhost:5173)
npm run build      # статическая сборка в build/ — adapter-static, ВСЁ пререндерится
npm run preview    # отдать собранное локально
npm run check      # svelte-check: типы в .svelte-файлах
npm run typecheck  # tsc --noEmit: чистый TS (ядро, модель, тесты)
```

Особенности:
- **adapter-static + `prerender = true`** (`src/routes/+layout.ts`): весь сайт — готовый HTML.
  Это осознанный ответ на боль индексации из `GOAL.md`, а не заглушка. Пререндер **валидирует
  внутренние ссылки и якоря** — битый `#anchor` роняет сборку (и это хорошо).
- Адаптер конфигурируется в `vite.config.ts` (новый SvelteKit не использует `svelte.config.js`).
- `tsconfig.json` расширяет генерируемый `.svelte-kit/tsconfig.json`; генерирует его
  `svelte-kit sync` (скрипт `prepare`, выполняется при `npm install`).
- **Node 24 исполняет TypeScript нативно** (стирание типов): ядро и его тесты не требуют сборки,
  импорты пишутся с расширением `.ts`.

### Зелёные тесты сами по себе ничего не доказывают

Прежде чем поверить набору тестов, **сломай код намеренно** и убедись, что они падают. На ядре это
проделано: мутации «показатель `0.7` → `1.0`», «произведение → среднее», «нет общих осей → связь всё
равно есть», «диаметр линеен вместо корня» дали 3, 6, 4 и 7 падений соответственно. Тест, который не
падает на сломанном коде, — не тест, а украшение. То же относится к любому смягчению правила в
`tools/03-kaif-verify.mjs`.

### Инварианты ядра (покрыты тестами)

- `|K| = 0` (нет общих осей) → связь не создаётся, возвращается `null`;
- человек сам с собой → `similarity = 100`;
- `d = D` (максимально далеко) → `proximity = 0` → `similarity = 0`;
- `similarity = proximity × commonality`, ни один множитель не компенсирует другой;
- симметрия главных величин; независимость от порядка осей;
- **похожесть считается на долях до округления**, а не как произведение округлённых процентов
  (разница в 1 п.п. ломает паритет с боевыми данными 1.x).

## Беклог и DONE-тег

Чтобы одного листинга файлов хватало понять, что открыто, а что закрыто, — **вставляй слово `DONE` в
имя файла после номера, когда задача файла выполнена и проверена:**

```
bugs/04_modal.md          →  bugs/04_DONE_modal.md
ideas/07_dev_menu.md      →  ideas/07_DONE_dev_menu.md
```

**Правило (делай это каждый раз, работая с файлами багов и идей):**
- Закончил баг или идею, и она ПОДТВЕРЖДЁННО закрыта (статус ✅, проверено) — переименуй немедленно,
  вставив `DONE` после номера: `git mv <NN>_<name>.md <NN>_DONE_<name>.md`.
- Файл в работе / частично / только исследование — НЕ помечай `DONE` (🔧/🟡/🔬 = ещё не готово).
- Используй `git mv` (сохраняет историю). Не меняй номер.
- Справочные документы (`MASTER_PLAN.md`, карты, `researches/`) — НЕ задачи, никогда не помечай их DONE.

**Скилл ревизии беклога — `/check-backlog`:** обходит `bugs/`, `ideas/` и `plans/`, собирает всё без
тега `DONE` как открытый беклог и помечает по-настоящему закрытые файлы (дописывая секцию статуса).

**Скилл заведения бага — `/report-bug`:** наткнулся на дефект при разработке или тесте — заведи
отдельный md в `bugs/` по канону, согласно `BUG_FIXING_FRAMEWORK.md`. Агент ведёт собственный беклог
багов — один документ на дефект, ничего не теряется.

**Скилл предложения идеи — `/propose-idea`:** появилась стоящая идея, укладывающаяся в мастер-план и
видение человека — заведи её как md в `ideas/` со статусом «❓ ожидает одобрения человека». Идея
